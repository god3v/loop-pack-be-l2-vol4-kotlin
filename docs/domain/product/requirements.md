# 상품(Product) 도메인 요구사항 명세

> **상태**: **v0.4 초안**
> **작성일**: 2026-05-19
> **변경 이력**:
> - v0.4 (2026-06-09): 브랜드 유스케이스·규칙(v0.3 의 UC-3~8)을 [brand/requirements.md](../brand/requirements.md) 로 **분리**한다(브랜드 명세에서 UC-1~6 으로 재정립). 본 명세는 회원 카탈로그(UC-1·UC-2) 와 관리자 상품 동작(UC-9~13) 을 다루며, **상품 UC 번호는 유지**하므로 v0.3 시절 브랜드가 쓰던 UC-3~8 자리는 비어 있다([api-spec.md](./api-spec.md) UC 참조와 정합). 관리자 인증은 [admin/logical-model.md](../admin/logical-model.md) 로 확정된다 — 헤더 `X-Loopers-Ldap: loopers.admin` 으로 식별하며 **별도 도메인 모델/계정을 두지 않는다**(v0.3 의 "관리자는 외부 액터" 서술 유지).
> - v0.3 (2026-05-20): 관리자(admin) 채널 추가 — 브랜드/상품의 등록·수정·삭제·목록·상세 유스케이스(UC-4 ~ UC-13). 삭제는 **soft delete**, 브랜드 삭제는 그 브랜드 상품의 **카스케이드 삭제(단일 트랜잭션)**, 상품의 브랜드는 등록 이후 **변경 불가**, 상품 이름은 **같은 브랜드 안에서 유일**, 상품 삭제 시 좋아요 행은 **유지(내 좋아요 목록에서 가려짐)**. **판매 상태(SalesStatus)** 가 도메인에 도입되어 관리자 응답에 포함된다. 관리자 인증은 본 명세에서 추상화(LDAP 등 구체 메커니즘은 logical-model/api-spec 위임). v0.2 의 "상품 판매 상태" TBD 는 도메인 도입 사실로 부분 해소(상태값 집합·회원 카탈로그 노출 정책은 신규 TBD 로 이월).
> - v0.2 (2026-05-19): 상품 목록 쿼리 파라미터 정렬: 정렬 허용 집합을 `latest` · `price_asc` · `likes_desc` 3개로 한정(표기는 snake_case). 페이지 기본값을 `page=0` / `size=20` 으로 확정. 미해결 결정 사항에서 "페이지 기본값" 항목을 해소하고 "페이지 크기 상한" 만 남긴다.
> - v0.1 (2026-05-19): 최초 초안. 카탈로그 3개 유스케이스(목록·상세·단일 브랜드 조회) 와 좋아요(외부 도메인) 결과의 반영 책임을 정의한다. 좋아요 등록/취소 유스케이스는 별도 `like` 도메인 명세에 위임한다.

## 1. 유비쿼터스 언어

| 용어 | 의미 |
|---|---|
| **상품(Product)** | 카탈로그에 노출되는 거래 단위. 정확히 하나의 브랜드에 속한다. 영문 표기는 `Product`. |
| **브랜드(Brand)** | 상품의 소속. 본 명세에서는 상품의 속성(`brandId`·브랜드 이름) 으로 등장하며, 브랜드 자체의 조회·관리는 [brand/requirements.md](../brand/requirements.md) 가 다룬다. |
| **카탈로그(Catalog)** | 본 도메인이 회원·비회원에게 제공하는 공개 조회 영역 — 상품 목록 · 상품 상세. |
| **좋아요(Like)** | 회원과 상품의 관계로, 회원이 상품에 표시하는 관심. **외부 도메인(`like`) 의 개념** 이며 본 도메인은 결과만 반영한다. |
| **좋아요 수(likeCount)** | 한 상품에 누적된 좋아요의 개수. 음수가 될 수 없다. |
| **내 좋아요 여부(likedByMe)** | 인증된 회원이 해당 상품에 좋아요를 표시한 상태인지 여부. 비회원/미인증 요청에서는 항상 거짓으로 응답된다. |
| **정렬 옵션(Sort)** | 상품 목록의 노출 순서를 결정하는 값. `latest` · `price_asc` · `likes_desc` 세 값으로 한정된다. 미지정 시 `latest` 가 적용된다. |
| **페이지(Page)** | 목록의 분할 단위. 페이지 번호(`page`, 0부터)와 페이지 크기(`size`) 두 값으로 식별되는 offset 페이징을 사용한다. 미지정 시 `page=0` / `size=20` 이 적용된다. |
| **브랜드 필터(brandId)** | 상품 목록 조회 시 특정 브랜드 소속만 노출하도록 지정하는 선택 입력. |
| **헤더 인증** | 회원(User) 도메인이 정의한 `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 기반 식별 절차. 본 도메인의 카탈로그 조회에는 필수가 아니나, 인증되면 `likedByMe` 가 의미를 가진다. |
| **관리자(Administrator)** | 상품의 등록·수정·삭제·조회를 수행하는 운영 주체. **외부 액터이며 도메인 모델로 등장하지 않는다** — 헤더(`X-Loopers-Ldap`)로 식별되며, 메커니즘은 [admin/logical-model.md](../admin/logical-model.md) 가 정의한다. |
| **관리자 채널** | `api-admin` 으로 노출되는 관리자 전용 경로 묶음. 회원 카탈로그 채널과 인증 방식·노출 정책이 분리된다. |
| **삭제 마크(Soft Delete)** | 상품을 도메인적으로 "존재하지 않는 것" 으로 다루기 위한 표시. 행은 보존되며, 회원 카탈로그와 관리자 목록·상세 모두에서 노출되지 않는다. |
| **판매 상태(SalesStatus)** | 상품의 현재 거래 가능성을 표현하는 값. 관리자 응답에 포함되며, 회원 카탈로그 노출 정책에 영향을 줄 수 있다. |

## 2. 유저 스토리

**회원 카탈로그**
- 사용자는 상품 목록을 조회할 수 있다 — 정렬 옵션과 브랜드 필터를 지정할 수 있다.
- 사용자는 상품 상세를 조회할 수 있다 — 좋아요 수가 함께 보인다.
- 인증된 회원은 상품 상세에서 자신이 그 상품을 좋아요했는지 함께 본다.
- 사용자는 인증 없이도 카탈로그(목록·상세) 를 조회할 수 있다.

**관리자**
- 관리자는 상품을 등록할 수 있다 — 이미 등록된 브랜드에 한해서 등록한다.
- 관리자는 상품 정보를 수정할 수 있다 — 단, 브랜드는 수정할 수 없다.
- 관리자는 상품을 삭제할 수 있다.
- 관리자는 등록된 상품 목록을 브랜드 필터·페이지 단위로 조회할 수 있다 — 최신순으로 본다.
- 관리자는 상품 상세를 조회할 수 있다 — 응답에 판매 상태가 포함된다.
- 관리자는 삭제된 상품을 목록·상세 어디에서도 보지 않는다.

> 브랜드 등록·수정·삭제·목록·상세 스토리는 [brand/requirements.md](../brand/requirements.md) 로 이동했다.

## 3. 기능 요구사항

**회원 카탈로그**
- 사용자가 페이지 번호 · 페이지 크기 · 정렬 옵션을 담아 상품 목록을 요청하면, 정렬된 페이지 단위 결과를 받는다.
- 사용자가 정렬 옵션을 지정하지 않으면 최신순(`latest`) 으로 응답된다.
- 사용자가 페이지 번호 또는 페이지 크기를 지정하지 않으면 기본값(`page=0` / `size=20`) 으로 응답된다.
- 사용자가 브랜드 필터를 지정하면 해당 브랜드 소속 상품만 응답된다.
- 사용자가 페이지 크기 또는 정렬 옵션을 허용 범위 밖으로 보내면 실패 응답을 받는다.
- 사용자가 상품 상세를 요청하면 상품 정보와 함께 좋아요 수를 응답받는다.
- 인증된 회원이 상품 상세를 요청하면 자신의 `likedByMe` 가 함께 응답된다 — 비회원/미인증 요청은 `likedByMe` 가 항상 거짓이다.
- 사용자가 존재하지 않는 상품 식별자로 상세를 요청하면 "찾을 수 없음" 실패 응답을 받는다.
- 좋아요 액션(별도 도메인) 의 결과는 이후의 카탈로그 응답에서 좋아요 수와 `likedByMe` 에 반영된다.
- 삭제 마크된 상품은 회원 카탈로그의 어느 응답에도 노출되지 않는다.

**관리자**
- 관리자 인증을 통과하지 않은 요청은 어떤 관리자 동작도 진행할 수 없다.
- 관리자가 상품을 등록하면 새 상품이 만들어진다 — 지정한 브랜드가 존재하지 않거나 삭제 마크되어 있으면 실패한다. 같은 브랜드 안에 같은 이름의 상품이 이미 있으면 실패한다.
- 관리자가 상품을 수정하면 변경된 정보로 갱신된다 — 브랜드는 수정 대상에서 제외된다.
- 관리자가 상품을 삭제하면 해당 상품이 삭제 마크 처리된다 — 그 상품에 등록된 좋아요 행은 유지되며, 회원의 내 좋아요 목록에서는 "삭제된 상품" 으로 가려진다.
- 관리자가 상품 목록을 요청하면 삭제 마크되지 않은 상품들을 최신순·페이지 단위로 받는다 — 브랜드 필터를 지정할 수 있다.
- 관리자가 보는 상품 정보(목록·상세) 에는 판매 상태가 포함된다.
- 관리자가 삭제 마크된 식별자로 조회·수정·삭제를 시도하면 "찾을 수 없음" 으로 응답된다.

## 4. 유스케이스 흐름

> 브랜드 유스케이스는 [brand/requirements.md](../brand/requirements.md) 로 분리됐다(거기서 UC-1~6 으로 재정립). 본 문서는 상품 UC 번호를 유지하므로 v0.3 시절 브랜드가 쓰던 UC-3~8 자리가 비어 있다.

### 회원 카탈로그

> UC-1, UC-2 는 인증을 **필수로 하지 않는다**. 인증 헤더가 함께 오면 회원 식별이 시도되며, 식별에 실패하면 카탈로그 응답은 비인증 응답과 동일하게 진행된다(어떤 카탈로그 유스케이스도 인증 실패로 거부되지 않는다).
> 본 절의 모든 유스케이스는 삭제 마크된 상품을 노출하지 않는다.

#### UC-1. 상품 목록 조회

**Main Flow**
1. 사용자가 페이지 번호(선택), 페이지 크기(선택), 정렬 옵션(선택), 브랜드 필터(선택) 를 담아 상품 목록을 요청한다.
2. 시스템은 페이지 번호 · 페이지 크기 · 정렬 옵션이 허용 범위·집합 안에 있는지 검증한다.
3. 시스템은 미지정 입력을 기본값으로 대체한다 — 정렬 옵션은 `latest`, 페이지 번호는 `0`, 페이지 크기는 `20`.
4. 시스템은 브랜드 필터가 주어졌으면 해당 브랜드 소속 상품만, 아니면 전체 상품을 정렬 순서대로 페이지 단위로 응답한다.
5. 응답 각 항목은 상품 식별자 · 이름 · 가격 · 좋아요 수 · 브랜드 요약 정보를 포함한다.

**Alternate Flow — 브랜드 필터 지정** (1 → 4a)
- 4a. 시스템은 브랜드 필터에 해당하는 상품만 같은 정렬 순서로 응답한다.

**Exception Flow**
- **E1** (2): 페이지 크기가 허용 범위를 벗어남 → `BAD_REQUEST` (400), 상태 변경 없음.
- **E2** (2): 정렬 옵션이 허용 집합(`latest` / `price_asc` / `likes_desc`) 에 없음 → `BAD_REQUEST` (400), 상태 변경 없음.
- **E3** (2): 페이지 번호가 음수 → `BAD_REQUEST` (400), 상태 변경 없음.

#### UC-2. 상품 상세 조회

**Main Flow**
1. 사용자가 상품 식별자로 상세를 요청한다.
2. 시스템은 해당 상품의 존재를 확인한다 — 삭제 마크된 상품은 존재하지 않는 것으로 다룬다.
3. 시스템은 상품 정보(식별자 · 이름 · 가격 · 설명 · 브랜드 정보 · 좋아요 수) 와 `likedByMe` 를 응답한다 — 이때 `likedByMe` 는 비회원/미인증 요청에서는 거짓으로 채워진다.

**Alternate Flow — 인증된 회원의 요청** (1 → 3a)
- 1a. 시스템은 헤더 인증이 함께 온 경우 회원을 식별한다.
- 3a. 시스템은 식별된 회원의 좋아요 여부를 조회해 `likedByMe` 에 채운다.

**Exception Flow**
- **E1** (2): 상품 식별자가 존재하지 않거나 삭제 마크되어 있음 → `PRODUCT_NOT_FOUND` (404), 상태 변경 없음.

### 관리자

> UC-9 ~ UC-13 은 모두 사전 단계로 **관리자 인증** 을 통과해야 한다. 관리자 인증의 구체 메커니즘은 [admin/logical-model.md](../admin/logical-model.md) 가 정의한다(전용 헤더 기반). 관리자 목록·상세 응답은 삭제 마크된 자원을 노출하지 않는다.

#### 공통 사전 단계 — 관리자 인증

1. 시스템은 요청에서 관리자 인증 정보를 확인한다.
2. 시스템은 관리자 인증 통과 여부를 판단한다.
3. 통과하면 해당 관리자를 컨텍스트로 두고 본 흐름으로 진입한다.

**Exception**
- **E0** (사전 단계 어디서든): 관리자 인증 실패 → `UNAUTHORIZED` (401), 상태 변경 없음.

#### UC-9. (관리자) 상품 목록 조회

**Main Flow**
1. 관리자가 페이지 번호(선택) · 페이지 크기(선택) · 브랜드 필터(선택) 를 담아 상품 목록을 요청한다.
2. 시스템은 페이지 번호 · 페이지 크기가 허용 범위 안에 있는지 검증한다.
3. 시스템은 미지정 입력을 기본값(`page=0` / `size=20`) 으로 대체한다.
4. 시스템은 삭제 마크되지 않은 상품들을 **최신순** 으로 페이지 단위로 응답한다.
5. 응답 각 항목은 상품 식별자 · 이름 · 가격 · 좋아요 수 · 브랜드 요약 · **판매 상태** 를 포함한다.

**Alternate Flow — 브랜드 필터 지정** (1 → 4a)
- 4a. 시스템은 브랜드 필터에 해당하는 상품만 같은 순서로 응답한다.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 페이지 번호가 음수이거나 페이지 크기가 허용 범위 밖 → `BAD_REQUEST` (400), 상태 변경 없음.

#### UC-10. (관리자) 상품 상세 조회

**Main Flow**
1. 관리자가 상품 식별자로 상세를 요청한다.
2. 시스템은 해당 상품의 존재를 확인한다 — 삭제 마크된 상품은 존재하지 않는 것으로 다룬다.
3. 시스템은 상품 정보(식별자 · 이름 · 가격 · 설명 · 브랜드 정보 · 좋아요 수 · **판매 상태**) 를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 상품 식별자가 존재하지 않거나 삭제 마크되어 있음 → `PRODUCT_NOT_FOUND` (404), 상태 변경 없음.

#### UC-11. (관리자) 상품 등록

**Main Flow**
1. 관리자가 브랜드 식별자 · 이름 · 가격 · 부가 정보(설명 등) 를 담아 상품 등록을 요청한다.
2. 시스템은 입력 필드의 필수·형식 규칙을 검증한다.
3. 시스템은 지정된 브랜드가 존재하고 삭제 마크되지 않았는지 확인한다.
4. 시스템은 같은 브랜드 안에 같은 이름의 상품이 존재하는지 확인한다.
5. 시스템은 상품을 새로 등록한다 — 판매 상태의 초깃값이 적용된다.
6. 시스템은 등록된 상품 식별자와 정보를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 입력이 필수·형식 규칙을 어김 → `PRODUCT_BAD_REQUEST` (400), 등록 없음.
- **E2** (3): 지정한 브랜드가 존재하지 않거나 삭제 마크되어 있음 → `BRAND_NOT_FOUND` (404), 등록 없음.
- **E3** (4): 같은 브랜드 안에 같은 이름의 상품이 이미 존재 → `DUPLICATE_PRODUCT_NAME` (409), 등록 없음.

#### UC-12. (관리자) 상품 정보 수정

**Main Flow**
1. 관리자가 상품 식별자와 변경할 정보(이름 · 가격 · 설명 · 판매 상태 등) 를 담아 수정을 요청한다 — **`brandId` 는 수정 입력에 받지 않는다**.
2. 시스템은 해당 상품의 존재를 확인한다 — 삭제 마크된 상품은 존재하지 않는 것으로 다룬다.
3. 시스템은 변경 입력의 필수·형식 규칙을 검증한다.
4. 이름이 변경되는 경우 시스템은 같은 브랜드 안에 같은 이름의 다른 상품이 있는지 확인한다 — 자기 자신은 중복 대상에서 제외된다.
5. 시스템은 상품 정보를 갱신한다.
6. 시스템은 갱신된 상품 정보를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 상품 식별자가 존재하지 않거나 삭제 마크되어 있음 → `PRODUCT_NOT_FOUND` (404), 변경 없음.
- **E2** (3): 변경 입력이 필수·형식 규칙을 어김 → `PRODUCT_BAD_REQUEST` (400), 변경 없음.
- **E3** (4): 같은 브랜드 안에 같은 이름의 다른 상품이 존재 → `DUPLICATE_PRODUCT_NAME` (409), 변경 없음.

#### UC-13. (관리자) 상품 삭제

**Main Flow**
1. 관리자가 상품 식별자로 삭제를 요청한다.
2. 시스템은 해당 상품의 존재를 확인한다 — 삭제 마크된 상품은 존재하지 않는 것으로 다룬다.
3. 시스템은 상품을 삭제 마크 처리한다 — 좋아요 행은 함께 제거하지 않는다.
4. 시스템은 빈 페이로드(`data: null`) 와 성공 응답을 돌려준다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 상품 식별자가 존재하지 않거나 이미 삭제 마크되어 있음 → `PRODUCT_NOT_FOUND` (404), 상태 변경 없음.

## 5. 도메인 규칙 / 불변식

**기본 불변식**
- 상품은 정확히 하나의 브랜드에 속한다 — 상품이 브랜드 없이 존재하지 않는다.
- 상품의 가격은 음수가 될 수 없다.
- 상품의 좋아요 수는 음수가 될 수 없다.

**유일성**
- 상품 이름은 같은 브랜드 안에서, 삭제 마크되지 않은 상품 사이에서 유일하다 — 다른 브랜드의 같은 이름은 허용된다.
- (브랜드 이름의 전체 유일성은 [brand/requirements.md](../brand/requirements.md) §5 참조.)

**브랜드 불변**
- 상품의 브랜드는 등록 이후 변경되지 않는다 — 본 도메인은 상품 수정 입력에서 `brandId` 를 받지 않는다.
- 상품 등록 시 지정된 브랜드는 이미 존재하며 삭제 마크되지 않아야 한다.

**삭제 마크(Soft Delete)**
- 본 도메인의 모든 삭제는 행 제거가 아니라 삭제 마크다 — 행 자체는 보존된다.
- 삭제 마크된 상품은 회원 카탈로그와 관리자 목록·상세 어디에서도 노출되지 않는다 — 존재 확인에서는 "찾을 수 없음" 으로 다룬다.
- 브랜드가 삭제되면 그 브랜드 소속 상품들도 함께 삭제 마크된다 — 카스케이드 규칙과 트랜잭션 경계는 [brand/requirements.md](../brand/requirements.md) UC-6·§5 가 소유한다.
- 상품 삭제는 그 상품에 대한 좋아요 행을 제거하지 않는다 — 좋아요 행은 유지되며, `like` 도메인의 내 좋아요 목록에서 "삭제된 상품" 으로 가려진다.

**카탈로그·정렬·페이징**
- 상품 목록의 정렬 옵션은 `latest` · `price_asc` · `likes_desc` 세 값 중 하나로 한정된다 — 회원 카탈로그(UC-1) 에만 적용된다. 미지정 입력은 `latest` 로 해석된다.
- 관리자 목록(UC-9) 의 정렬은 **최신순 고정** 이다 — 정렬 옵션 입력을 받지 않는다.
- 페이지 번호의 기본값은 `0`, 페이지 크기의 기본값은 `20` 이다. 미지정 입력은 기본값으로 해석된다.
- 페이지 크기 · 페이지 번호 · 정렬 옵션은 허용 범위/집합 밖이면 실패 응답으로 분기된다 — 조용한 보정·기본값 대체는 미지정의 경우에만 일어난다.
- 존재하지 않는 상품 식별자(또는 삭제 마크된 식별자) 로의 조회·수정·삭제는 일반 응답으로 위장하지 않고 별도의 "찾을 수 없음" 실패로 분기된다.

**좋아요 결합**
- `likedByMe` 는 인증된 회원에 대해서만 의미를 가진다 — 비회원/미인증 요청에서는 항상 거짓이다.
- 본 도메인은 좋아요의 등록·취소 책임을 지지 않는다 — 그 책임은 별도 `like` 도메인에 있다.
- 본 도메인은 좋아요 액션의 결과를 카탈로그 응답(좋아요 수, `likedByMe`) 에 반영하는 의무를 진다.
- 좋아요는 외부 도메인의 약속에 따라 멱등 동작이다 — 본 도메인은 그 멱등 결과가 자신의 응답에 일관되게 보이도록 한다.

**판매 상태**
- 상품은 **판매 상태(SalesStatus)** 를 가진다 — 관리자 응답(목록·상세) 에 포함된다.
- 상품 등록 시 판매 상태에는 초깃값이 부여된다.
- **TBD**: 판매 상태의 전이 규칙. (값 집합은 구현에서 `on_sale`/`out_of_stock`/`off_sale` 로 확정 — `api-spec.md` §0.3 참조.)
- **TBD**: 회원 카탈로그(UC-1, UC-2) 가 판매 상태에 따라 노출을 제한하거나 상태를 함께 노출하는지 — 별도 결정이다.

**인증 채널 분리**
- 회원 카탈로그(UC-1 ~ UC-2) 는 인증을 필수로 하지 않는다 — 인증 부재가 거부 사유가 되지 않는다.
- 관리자 동작(UC-9 ~ UC-13) 은 관리자 인증을 통과한 주체에게만 허용된다.
- 관리자 채널과 회원 채널은 노출 경로(`api-admin` vs `api`) 와 인증 방식이 분리된다.

> 데이터 구조 결정은 `logical-model.md` 를 참고한다 — 상품 전용 logical-model 은 아직 작성되지 않았으며, 관리자 인증의 데이터/메커니즘 결정은 [admin/logical-model.md](../admin/logical-model.md) 가 다룬다.
