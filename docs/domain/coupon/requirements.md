# 쿠폰(Coupon) 도메인 요구사항 명세

> **상태**: **v0.2 초안**
> **작성일**: 2026-06-09
> **변경 이력**:
> - v0.2 (2026-06-10): **주문 적용(UC-9) 을 범위에 포함**한다. 쿠폰은 **주문 1건당 1장**만 적용되고, 존재하지 않음/이미 사용/만료/타 유저 소유 쿠폰으로 주문하면 **주문이 실패**(쿠폰 소진·재고 차감 모두 롤백)하며, 주문 성공 시 해당 쿠폰은 **즉시 USED 로 전이**되어 재사용 불가다. v0.1 에서 "범위 밖" 으로 두었던 `OrderFacade` 연동을 실제 구현으로 반영한다(주문 생성 트랜잭션 안에서 쿠폰을 소진).
> - v0.1 (2026-06-09): 최초 초안. 쿠폰 **템플릿(정의)** 과 **발급 쿠폰(사용자 보유 인스턴스)** 을 분리한 2-애그리거트 모델로 정의한다. 회원 채널(발급 요청·내 목록 조회), 관리자 채널(템플릿 CRUD·발급 내역 조회), 그리고 주문에 적용되는 **쿠폰 사용(할인 계산 + 단일 사용 보장)** 의 도메인 책임을 다룬다. 발급은 **1인 1매**, 사용은 **최대 1회**, 만료 상태는 **조회·사용 시 파생**으로 합의했다.

---

## 1. 유비쿼터스 언어

| 용어 | 의미 |
|---|---|
| **쿠폰(Coupon)** | 관리자가 정의·관리하는 **할인 정책 템플릿**. 이름·할인 종류·할인 값·최소 주문 금액(선택)·만료 시각을 가진다. `POST /coupons/{couponId}/issue` 의 `couponId` 는 이 템플릿의 식별자다. 영문 표기는 `Coupon`. |
| **할인 종류(DiscountType)** | 쿠폰의 할인 계산 방식. **정액(`FIXED`)** / **정률(`RATE`)** 두 값으로 한정된다. |
| **할인 값(discountValue)** | 정액이면 할인 금액(원), 정률이면 할인 비율(%). 양수다. 정률은 1~100 범위다. |
| **최소 주문 금액(minOrderAmount)** | 쿠폰을 적용할 수 있는 주문 합계의 하한. 선택 입력이며, 미지정이면 하한 제약이 없다. |
| **만료 시각(expiredAt)** | 쿠폰이 더 이상 발급·사용될 수 없게 되는 절대 시각. |
| **발급 쿠폰(UserCoupon)** | 회원이 쿠폰 템플릿으로부터 발급받아 소유하는 **인스턴스**. 어느 회원이 어느 템플릿을 발급받았는지와 사용 여부를 가진다. 영문 표기는 `UserCoupon`. |
| **발급(Issue)** | 회원이 쿠폰 템플릿으로부터 자신의 발급 쿠폰 1매를 만드는 행위. |
| **사용(Use)** | 발급 쿠폰을 주문에 적용해 소진하는 행위. 발급 쿠폰은 최대 한 번 사용된다. |
| **발급 쿠폰 상태(UserCouponStatus)** | 발급 쿠폰의 노출 상태. **사용 가능(`AVAILABLE`)** / **사용 완료(`USED`)** / **만료(`EXPIRED`)** 셋으로 노출된다. 만료는 별도 저장값이 아니라 만료 시각 경과로 **파생**되는 노출 상태다. |
| **할인 금액(discountAmount)** | 쿠폰 사용 시 주문 합계에서 차감되는 금액. 음수가 될 수 없고, 주문 합계를 초과하지 않는다. |
| **회원(User)** | 쿠폰을 발급받고 보유 목록을 조회하며 주문에 사용하는 주체. `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더로 식별된다. |
| **관리자(Administrator)** | 쿠폰 템플릿을 등록·수정·삭제·조회하고 발급 내역을 조회하는 운영 주체. **외부 액터이며 도메인 모델로 등장하지 않는다** — 헤더(`X-Loopers-Ldap`)로 식별되며, 메커니즘은 [admin/logical-model.md](../admin/logical-model.md) 가 정의한다. |
| **관리자 채널** | `api-admin` 으로 노출되는 관리자 전용 경로 묶음. 회원 채널과 인증 방식·노출 정책이 분리된다. |
| **삭제 마크(Soft Delete)** | 쿠폰 템플릿을 도메인적으로 "존재하지 않는 것" 으로 다루기 위한 표시. 행은 보존되며, 관리자 목록·상세에서 노출되지 않는다. 이미 발급된 쿠폰의 조회·사용에는 영향을 주지 않는다. |

## 2. 유저 스토리

**회원**
- 회원은 쿠폰 템플릿으로부터 쿠폰을 발급받을 수 있다.
- 회원은 같은 쿠폰 템플릿을 한 번만 발급받을 수 있다.
- 회원은 자신이 보유한 쿠폰 목록을 사용 가능 / 사용 완료 / 만료 상태와 함께 조회할 수 있다.
- 회원은 주문 시 자신이 보유한 사용 가능한 쿠폰을 적용해 할인받을 수 있다 — 주문 1건당 한 장만 적용한다.
- 회원은 사용 불가능한 쿠폰(존재하지 않음 · 이미 사용 · 만료 · 타 유저 소유) 으로는 주문을 완료할 수 없다.

**관리자**
- 관리자는 쿠폰 템플릿을 등록할 수 있다 — 정액 또는 정률로 지정한다.
- 관리자는 쿠폰 템플릿을 수정할 수 있다.
- 관리자는 쿠폰 템플릿을 삭제할 수 있다.
- 관리자는 쿠폰 템플릿 목록을 페이지 단위로 조회할 수 있다.
- 관리자는 쿠폰 템플릿 상세를 조회할 수 있다.
- 관리자는 특정 쿠폰 템플릿의 발급 내역을 페이지 단위로 조회할 수 있다.

## 3. 기능 요구사항

사용자 관점에서 관찰 가능한 동작만 적는다.

**회원**
- 로그인한 회원이 쿠폰 템플릿 식별자로 발급을 요청하면, 자신의 발급 쿠폰 1매가 생기고 그 정보를 응답받는다.
- 로그인한 회원이 이미 발급받은 템플릿으로 다시 발급을 요청하면, 중복 발급 실패 응답을 받는다.
- 회원이 존재하지 않거나 삭제된 쿠폰 템플릿으로 발급을 요청하면, "찾을 수 없음" 실패 응답을 받는다.
- 회원이 이미 만료된 쿠폰 템플릿으로 발급을 요청하면, 사용 불가 실패 응답을 받는다.
- 인증 없이 발급을 요청하면, 인증 실패 응답을 받는다.
- 로그인한 회원이 내 쿠폰 목록을 요청하면, 자신이 보유한 발급 쿠폰들을 페이지 단위로 받는다 — 각 항목에 사용 가능 / 사용 완료 / 만료 상태가 함께 보인다.
- 회원이 주문에 사용 가능한 보유 쿠폰을 적용하면, 할인 금액이 계산되어 주문 합계에 반영되고 그 쿠폰은 **즉시 사용 완료** 가 되어 다시 사용할 수 없다.
- 한 주문에는 쿠폰을 한 장만 적용할 수 있다.
- 회원이 존재하지 않거나 자신의 소유가 아닌 쿠폰으로 주문하면, 주문이 실패한다.
- 회원이 이미 사용한 쿠폰으로 주문하면, 주문이 실패한다.
- 회원이 만료된 쿠폰으로 주문하면, 주문이 실패한다.
- 회원이 최소 주문 금액에 못 미치는 주문에 쿠폰을 적용하면, 주문이 실패한다.
- 쿠폰 적용이 실패하면 그 주문은 만들어지지 않으며, 차감된 재고와 쿠폰 사용도 함께 되돌려진다.

**관리자**
- 관리자 인증을 통과하지 않은 요청은 어떤 관리자 동작도 진행할 수 없다.
- 관리자가 쿠폰 템플릿을 등록하면 새 템플릿이 만들어진다 — 할인 종류(정액/정률), 할인 값, 만료 시각이 형식 규칙을 어기면 실패한다.
- 관리자가 쿠폰 템플릿을 수정하면 변경된 정보로 갱신된다.
- 관리자가 쿠폰 템플릿을 삭제하면 해당 템플릿이 삭제 마크 처리된다 — 이미 발급된 쿠폰은 그대로 유지된다.
- 관리자가 쿠폰 템플릿 목록을 요청하면 삭제 마크되지 않은 템플릿들을 페이지 단위로 받는다.
- 관리자가 쿠폰 템플릿 상세를 요청하면 해당 템플릿 정보를 받는다.
- 관리자가 삭제 마크된 식별자로 조회·수정·삭제를 시도하면 "찾을 수 없음" 으로 응답된다.
- 관리자가 특정 쿠폰 템플릿의 발급 내역을 요청하면 그 템플릿으로 발급된 발급 쿠폰들을 페이지 단위로 받는다 — 각 항목에 발급 회원과 사용 여부가 보인다.

## 4. 유스케이스 흐름

### 회원

> UC-1, UC-2 는 **회원 인증을 필수로 한다** — 헤더(`X-Loopers-LoginId`/`X-Loopers-LoginPw`)가 없거나 식별에 실패하면 거부된다.

#### UC-1. 쿠폰 발급 요청

**Main Flow**
1. 회원이 쿠폰 템플릿 식별자로 발급을 요청한다.
2. 시스템은 해당 템플릿의 존재를 확인한다 — 삭제 마크된 템플릿은 존재하지 않는 것으로 다룬다.
3. 시스템은 템플릿이 만료되지 않았는지 확인한다.
4. 시스템은 회원이 같은 템플릿을 이미 발급받았는지 확인한다.
5. 시스템은 발급 쿠폰 1매를 만들어 회원에게 귀속시킨다 — 초기 상태는 사용 가능이다.
6. 시스템은 발급된 쿠폰 정보를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- **E1** (2): 템플릿 식별자가 존재하지 않거나 삭제 마크되어 있음 → `COUPON_NOT_FOUND` (404), 발급 없음.
- **E2** (3): 템플릿이 이미 만료됨 → `COUPON_NOT_APPLICABLE` (400), 발급 없음.
- **E3** (4): 회원이 같은 템플릿을 이미 발급받음 → `ALREADY_ISSUED_COUPON` (409), 발급 없음.
- **E4** (1): 회원 인증 실패(헤더 누락/계정 없음/비번 불일치) → `UNAUTHORIZED` (401), 발급 없음.

#### UC-2. 내 쿠폰 목록 조회

**Main Flow**
1. 회원이 페이지 번호(선택) · 페이지 크기(선택) 를 담아 내 쿠폰 목록을 요청한다.
2. 시스템은 미지정 입력을 기본값(`page=0` / `size=20`) 으로 대체한다.
3. 시스템은 회원이 보유한 발급 쿠폰들을 발급 최신순으로 페이지 단위로 응답한다.
4. 응답 각 항목은 쿠폰 식별자 · 이름 · 할인 종류 · 할인 값 · 최소 주문 금액 · 만료 시각 · **노출 상태(사용 가능 / 사용 완료 / 만료)** 를 포함한다 — 노출 상태는 사용 여부와 만료 시각 경과를 합쳐 판정된다.

**Alternate Flow** — 없음.

**Exception Flow**
- **E1** (1): 회원 인증 실패 → `UNAUTHORIZED` (401), 상태 변경 없음.

#### UC-9. 쿠폰 사용(주문 적용)

> 본 유스케이스는 회원이 직접 호출하는 별도 엔드포인트가 아니라, **주문 생성 흐름이 본 도메인에 위임하는 능력** 이다. 주문 생성(`OrderFacade.placeOrder`) 트랜잭션 안에서 호출되며 — 쿠폰 적용 성공 시 발급 쿠폰은 같은 트랜잭션에서 즉시 소진(USED)되고 할인 금액이 주문 합계에 반영된다. **쿠폰은 주문 1건당 한 장만** 적용된다(주문은 발급 쿠폰 식별자 한 개만 입력으로 받는다). 적용이 실패하면 본 예외가 그대로 전파되어 **주문 전체가 롤백**된다(주문 미생성 · 재고 차감 원복 · 쿠폰 미소진).

**Main Flow**
1. 주문 흐름이 회원 · 발급 쿠폰 식별자(주문당 1개) · 주문 합계를 담아 쿠폰 적용을 요청한다.
2. 시스템은 발급 쿠폰의 존재와 그 회원 소유 여부를 확인한다.
3. 시스템은 발급 쿠폰이 아직 사용되지 않았는지 확인한다.
4. 시스템은 쿠폰 템플릿이 만료되지 않았는지 확인한다.
5. 시스템은 주문 합계가 최소 주문 금액 이상인지 확인한다.
6. 시스템은 할인 종류에 따라 할인 금액을 계산한다 — 정액은 할인 금액, 정률은 비율 적용 금액이며, 어느 경우든 주문 합계를 초과하지 않는다.
7. 시스템은 발급 쿠폰을 즉시 사용 완료로 전이시키고 할인 금액을 돌려준다 — 주문이 확정되면 이 소진이 함께 커밋된다.

**Alternate Flow** — 없음.

**Exception Flow** (모든 예외는 주문 흐름으로 전파되어 주문을 실패시키고, 같은 트랜잭션의 재고 차감·쿠폰 소진을 롤백한다)
- **E1** (2): 발급 쿠폰이 존재하지 않거나 그 회원의 소유가 아님 → `USER_COUPON_NOT_FOUND` (404), 주문 실패.
- **E2** (3): 발급 쿠폰이 이미 사용됨 → `ALREADY_USED_COUPON` (409), 주문 실패.
- **E3** (4): 쿠폰 템플릿이 만료됨 → `COUPON_NOT_APPLICABLE` (400), 주문 실패.
- **E4** (5): 주문 합계가 최소 주문 금액에 못 미침 → `COUPON_NOT_APPLICABLE` (400), 주문 실패.

---

### 관리자

> UC-3 ~ UC-8 은 모두 사전 단계로 **관리자 인증** 을 통과해야 한다. 관리자 인증의 구체 메커니즘은 [admin/logical-model.md](../admin/logical-model.md) 가 정의한다(전용 헤더 기반). 관리자 목록·상세 응답은 삭제 마크된 템플릿을 노출하지 않는다.

#### 공통 사전 단계 — 관리자 인증

1. 시스템은 요청에서 관리자 인증 정보를 확인한다.
2. 시스템은 관리자 인증 통과 여부를 판단한다.
3. 통과하면 본 흐름으로 진입한다.

**Exception**
- **E0** (사전 단계 어디서든): 관리자 인증 실패 → `UNAUTHORIZED` (401), 상태 변경 없음.

#### UC-3. (관리자) 쿠폰 템플릿 목록 조회

**Main Flow**
1. 관리자가 페이지 번호(선택) · 페이지 크기(선택) 를 담아 템플릿 목록을 요청한다.
2. 시스템은 미지정 입력을 기본값(`page=0` / `size=20`) 으로 대체한다.
3. 시스템은 삭제 마크되지 않은 템플릿들을 **최신순** 으로 페이지 단위로 응답한다.
4. 응답 각 항목은 템플릿 식별자 · 이름 · 할인 종류 · 할인 값 · 최소 주문 금액 · 만료 시각을 포함한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).

#### UC-4. (관리자) 쿠폰 템플릿 상세 조회

**Main Flow**
1. 관리자가 템플릿 식별자로 상세를 요청한다.
2. 시스템은 해당 템플릿의 존재를 확인한다 — 삭제 마크된 템플릿은 존재하지 않는 것으로 다룬다.
3. 시스템은 템플릿 정보(식별자 · 이름 · 할인 종류 · 할인 값 · 최소 주문 금액 · 만료 시각) 를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 템플릿 식별자가 존재하지 않거나 삭제 마크되어 있음 → `COUPON_NOT_FOUND` (404), 상태 변경 없음.

#### UC-5. (관리자) 쿠폰 템플릿 등록

**Main Flow**
1. 관리자가 이름 · 할인 종류 · 할인 값 · 최소 주문 금액(선택) · 만료 시각을 담아 등록을 요청한다.
2. 시스템은 입력의 필수·형식 규칙을 검증한다 — 할인 종류는 정액/정률 중 하나, 할인 값은 양수(정률이면 1~100), 만료 시각은 미래 시각이다.
3. 시스템은 쿠폰 템플릿을 새로 등록한다.
4. 시스템은 등록된 템플릿 식별자와 정보를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 입력이 필수·형식 규칙을 어김(이름/할인 종류/할인 값/만료 시각) → `COUPON_BAD_REQUEST` (400), 등록 없음.

#### UC-6. (관리자) 쿠폰 템플릿 수정

**Main Flow**
1. 관리자가 템플릿 식별자와 변경할 정보(이름 · 할인 종류 · 할인 값 · 최소 주문 금액 · 만료 시각) 를 담아 수정을 요청한다.
2. 시스템은 해당 템플릿의 존재를 확인한다 — 삭제 마크된 템플릿은 존재하지 않는 것으로 다룬다.
3. 시스템은 변경 입력의 필수·형식 규칙을 검증한다.
4. 시스템은 템플릿 정보를 갱신한다.
5. 시스템은 갱신된 템플릿 정보를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 템플릿 식별자가 존재하지 않거나 삭제 마크되어 있음 → `COUPON_NOT_FOUND` (404), 변경 없음.
- **E2** (3): 변경 입력이 필수·형식 규칙을 어김 → `COUPON_BAD_REQUEST` (400), 변경 없음.

#### UC-7. (관리자) 쿠폰 템플릿 삭제

**Main Flow**
1. 관리자가 템플릿 식별자로 삭제를 요청한다.
2. 시스템은 해당 템플릿의 존재를 확인한다 — 삭제 마크된 템플릿은 존재하지 않는 것으로 다룬다.
3. 시스템은 템플릿을 삭제 마크 처리한다 — 그 템플릿으로 이미 발급된 쿠폰은 제거하지 않는다.
4. 시스템은 빈 페이로드(`data: null`) 와 성공 응답을 돌려준다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 템플릿 식별자가 존재하지 않거나 이미 삭제 마크되어 있음 → `COUPON_NOT_FOUND` (404), 상태 변경 없음.

#### UC-8. (관리자) 특정 쿠폰의 발급 내역 조회

**Main Flow**
1. 관리자가 템플릿 식별자와 페이지 번호(선택) · 페이지 크기(선택) 를 담아 발급 내역을 요청한다.
2. 시스템은 미지정 입력을 기본값(`page=0` / `size=20`) 으로 대체한다.
3. 시스템은 해당 템플릿으로 발급된 발급 쿠폰들을 발급 최신순으로 페이지 단위로 응답한다.
4. 응답 각 항목은 발급 쿠폰 식별자 · 발급 회원 식별자 · 노출 상태(사용 가능 / 사용 완료 / 만료) · 발급 시각 · 사용 시각(있으면) 을 포함한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (1): 템플릿 식별자가 존재하지 않거나 삭제 마크되어 있음 → `COUPON_NOT_FOUND` (404), 상태 변경 없음.

## 5. 도메인 규칙 / 불변식

**쿠폰 템플릿**
- 쿠폰 템플릿은 정확히 하나의 할인 종류(정액/정률) 를 가진다.
- 할인 값은 양수다 — 정률이면 1~100(%) 범위 안이다.
- 최소 주문 금액은 선택이다 — 있으면 음수가 될 수 없고, 없으면 하한 제약이 없다.
- 만료 시각은 등록 시점 기준 미래 시각이다.
- 쿠폰 템플릿의 삭제는 행 제거가 아니라 삭제 마크다 — 삭제 마크된 템플릿은 관리자 목록·상세에서 노출되지 않으며, 존재 확인에서 "찾을 수 없음" 으로 다뤄진다.
- 쿠폰 템플릿의 삭제는 그 템플릿으로 이미 발급된 쿠폰을 제거하지 않는다 — 이미 발급된 쿠폰은 자신의 조건대로 조회·사용될 수 있다.

**발급(1인 1매)**
- 한 회원은 같은 쿠폰 템플릿으로부터 최대 한 매만 발급받는다 — (회원, 템플릿) 조합은 유일하다.
- 발급 쿠폰은 정확히 하나의 회원과 하나의 템플릿에 귀속된다.
- 발급은 만료되지 않은 템플릿에 대해서만 허용된다.

**사용(최대 1회)**
- 발급 쿠폰은 최대 한 번 사용된다 — 사용 완료된 쿠폰은 다시 사용할 수 없다.
- 쿠폰은 **주문 1건당 한 장**만 적용된다.
- 주문이 성공하면 적용된 쿠폰은 **즉시 사용 완료(USED)** 로 전이되며, 이 소진은 주문 생성과 **같은 트랜잭션**에서 일어난다 — 주문이 실패하면 소진도 함께 롤백되어 쿠폰은 다시 사용 가능 상태로 남는다.
- 같은 발급 쿠폰에 대한 동시 사용 시도는 둘 이상이 성공하지 않는다 — 단일 사용은 동시성 하에서도 보장된다.
- 만료된 쿠폰은 사용할 수 없다.
- 주문 합계가 최소 주문 금액에 못 미치면 쿠폰을 사용할 수 없다.
- 존재하지 않거나 타 유저 소유인 쿠폰으로는 사용할 수 없다 — 두 경우 모두 "보유 쿠폰을 찾을 수 없음" 으로 다뤄 존재 여부를 노출하지 않는다.

**할인 계산**
- 정액 쿠폰의 할인 금액은 할인 값(원) 이되, 주문 합계를 초과하지 않는다.
- 정률 쿠폰의 할인 금액은 `주문 합계 × 할인 값(%)` 을 버림 처리한 값이되, 주문 합계를 초과하지 않는다.
- 할인 금액은 음수가 될 수 없으며, 적용 후 결제 대상 금액은 0 이상이다.

**노출 상태(파생)**
- 발급 쿠폰의 노출 상태는 사용 가능 / 사용 완료 / 만료 셋 중 하나다.
- 만료는 별도로 저장되는 상태가 아니라, 사용되지 않은 발급 쿠폰의 템플릿 만료 시각이 현재 시각을 지났을 때 파생되는 노출 상태다.
- 사용 완료는 만료보다 우선한다 — 이미 사용된 쿠폰은 만료 시각을 지나도 사용 완료로 노출된다.

**인증 채널 분리**
- 회원 동작(UC-1 · UC-2) 은 회원 인증을 통과한 주체에게만 허용된다.
- 관리자 동작(UC-3 ~ UC-8) 은 관리자 인증을 통과한 주체에게만 허용된다.
- 관리자 채널과 회원 채널은 노출 경로(`api-admin` vs `api`) 와 인증 방식이 분리된다.

> 데이터 구조 결정은 [docs/design/04-erd.md](../../design/04-erd.md) 의 `coupons` · `user_coupons` 모델링을 참고한다. (쿠폰 전용 `logical-model.md` 는 두지 않고 전사 ERD 에 통합한다.)

## 6. 비기능 요구사항

- 같은 회원이 같은 템플릿을 동시에 발급 요청해도 1인 1매 약속을 위반하지 않는다.
- 같은 발급 쿠폰에 대한 동시 사용 요청이 있어도 최대 1회 사용 약속을 위반하지 않는다.
- 모든 응답은 표준 응답 봉투(`ApiResponse<T>`) 로 감싼다.
- 만료 시각·발급 시각·사용 시각 등 시간 비교는 `Asia/Seoul` 기준으로 일관되게 처리한다.

## 미해결 결정 사항

- (현재 없음.) 주문 적용(UC-9) 은 `OrderFacade.placeOrder` 트랜잭션 안에서 본 도메인의 쿠폰 사용 능력을 호출하는 방식으로 **구현됐다**. 현재 주문 흐름에는 별도 결제 단계가 없어(주문은 `PAYMENT_PENDING` 생성으로 성공), 결제 실패 시의 쿠폰 사용 보상은 결제 단계가 도입될 때 다룬다 — 그때까지는 주문 생성 트랜잭션 실패가 곧 쿠폰 소진 롤백이다.
- 동시성 단일 사용(§5 · §6) 은 **사용 경로의 발급 쿠폰 행 비관적 쓰기 락(`SELECT ... FOR UPDATE`)** 으로 보장한다 — 같은 발급 쿠폰을 다투는 동시 주문은 직렬화되어 정확히 한 건만 소진에 성공하고 나머지는 `ALREADY_USED_COUPON` 으로 실패한다(`OrderCouponIntegrationTest` 동시성 케이스로 회귀 검증). 재고 차감 등 본 도메인 밖의 동시성 정책은 별개다.
