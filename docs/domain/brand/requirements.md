# 브랜드(Brand) 도메인 요구사항 명세

> **상태**: v0.1 초안 (상품 요구사항 v0.3 에서 분리)
> **작성일**: 2026-06-09
> **변경 이력**:
> - v0.1 (2026-06-09): 상품(Product) 요구사항 v0.3 에서 브랜드 유스케이스와 도메인 규칙을 분리해 독립 명세로 수립한다. 회원 단일 조회(UC-1) 와 관리자 브랜드 동작(UC-2~6) 을 다룬다. 독립 명세이므로 **유스케이스 번호를 UC-1 부터 재정립**한다(통합 명세 v0.3 의 UC-3~8 → 본 명세 UC-1~6). 관리자 인증의 구체 메커니즘은 [admin/logical-model.md](../admin/logical-model.md) 가 정의한다.

---

## 1. 유비쿼터스 언어

| 용어 | 의미 |
|---|---|
| **브랜드(Brand)** | 상품의 소속. 단일 정보 조회의 주체이며, 관리자 등록·수정·삭제의 대상이다. 영문 표기는 `Brand`. |
| **상품(Product)** | 브랜드에 속하는 거래 단위. 본 명세에서는 브랜드 삭제가 동반 삭제하는 **소속 자원**으로 등장한다. 상세는 [product/requirements.md](../product/requirements.md). |
| **관리자(Administrator)** | 브랜드의 등록·수정·삭제·조회를 수행하는 운영 주체. **외부 액터이며 도메인 모델로 등장하지 않는다** — 헤더(`X-Loopers-Ldap`)로 식별되며, 메커니즘은 [admin/logical-model.md](../admin/logical-model.md) 가 정의한다. |
| **관리자 채널** | `api-admin` 으로 노출되는 관리자 전용 경로 묶음. 회원 카탈로그 채널과 인증 방식·노출 정책이 분리된다. |
| **삭제 마크(Soft Delete)** | 브랜드를 도메인적으로 "존재하지 않는 것" 으로 다루기 위한 표시. 행은 보존되며, 회원·관리자 조회 모두에서 노출되지 않는다. |
| **카스케이드 삭제** | 브랜드 삭제 마크 처리에 그 브랜드 소속 상품들의 삭제 마크 처리가 동반되는 규칙. 같은 트랜잭션 경계 안에서 이루어진다. |
| **페이지(Page)** | 관리자 목록의 분할 단위. 페이지 번호(`page`, 0부터)와 페이지 크기(`size`) 두 값으로 식별되는 offset 페이징을 사용한다. 미지정 시 `page=0` / `size=20` 이 적용된다. |

---

## 2. 유저 스토리

**회원**
- 사용자는 단일 브랜드 정보를 조회할 수 있다 — 인증 없이도 조회할 수 있다.

**관리자**
- 관리자는 브랜드를 등록할 수 있다.
- 관리자는 브랜드 정보를 수정할 수 있다.
- 관리자는 브랜드를 삭제할 수 있다 — 그 브랜드의 상품들도 함께 삭제 처리된다.
- 관리자는 등록된 브랜드 목록을 페이지 단위로 조회할 수 있다.
- 관리자는 단일 브랜드 상세를 조회할 수 있다.
- 관리자는 삭제된 브랜드를 목록·상세 어디에서도 보지 않는다.

---

## 3. 기능 요구사항

**회원**
- 사용자가 브랜드 식별자로 단일 브랜드를 요청하면 브랜드 정보를 응답받는다.
- 사용자가 존재하지 않는(또는 삭제 마크된) 브랜드 식별자로 조회하면 "찾을 수 없음" 실패 응답을 받는다.

**관리자**
- 관리자 인증을 통과하지 않은 요청은 어떤 관리자 동작도 진행할 수 없다.
- 관리자가 브랜드를 등록하면 새 브랜드가 만들어진다 — 이름이 이미 존재하는 브랜드와 중복되면 실패한다.
- 관리자가 브랜드를 수정하면 변경된 정보로 갱신된다 — 이름이 다른 브랜드와 중복되면 실패한다.
- 관리자가 브랜드를 삭제하면 해당 브랜드와 그 브랜드 소속 상품들이 함께 삭제 마크 처리된다.
- 관리자가 브랜드 목록을 요청하면 삭제 마크되지 않은 브랜드들을 최신순·페이지 단위로 받는다.
- 관리자가 브랜드 상세를 요청하면 브랜드 정보를 받는다 — 삭제 마크된 브랜드는 "찾을 수 없음" 으로 응답된다.

## 4. 유스케이스 흐름

### 회원 카탈로그

> UC-1 은 인증을 **필수로 하지 않는다**. 삭제 마크된 브랜드는 노출하지 않는다.

#### UC-1. 단일 브랜드 조회

**Main Flow**
1. 사용자가 브랜드 식별자로 단일 브랜드 조회를 요청한다.
2. 시스템은 해당 브랜드의 존재를 확인한다 — 삭제 마크된 브랜드는 존재하지 않는 것으로 다룬다.
3. 시스템은 브랜드 정보(식별자 · 이름 · 설명 등) 를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- **E1** (2): 브랜드 식별자가 존재하지 않거나 삭제 마크되어 있음 → `BRAND_NOT_FOUND` (404), 상태 변경 없음.

---

### 관리자

> UC-2 ~ UC-6 은 모두 사전 단계로 **관리자 인증** 을 통과해야 한다. 관리자 인증의 구체 메커니즘은 [admin/logical-model.md](../admin/logical-model.md) 가 정의한다(전용 헤더 기반). 관리자 목록·상세 응답은 삭제 마크된 자원을 노출하지 않는다.

#### 공통 사전 단계 — 관리자 인증

1. 시스템은 요청에서 관리자 인증 정보를 확인한다.
2. 시스템은 관리자 인증 통과 여부를 판단한다.
3. 통과하면 해당 관리자를 컨텍스트로 두고 본 흐름으로 진입한다.

**Exception**
- **E0** (사전 단계 어디서든): 관리자 인증 실패 → `UNAUTHORIZED` (401), 상태 변경 없음.

#### UC-2. (관리자) 브랜드 목록 조회

**Main Flow**
1. 관리자가 페이지 번호(선택) · 페이지 크기(선택) 를 담아 브랜드 목록을 요청한다.
2. 시스템은 페이지 번호 · 페이지 크기가 허용 범위 안에 있는지 검증한다.
3. 시스템은 미지정 입력을 기본값(`page=0` / `size=20`) 으로 대체한다.
4. 시스템은 삭제 마크되지 않은 브랜드들을 **최신순** 으로 페이지 단위로 응답한다.
5. 응답 각 항목은 브랜드 식별자 · 이름 · 설명 등 브랜드 요약을 포함한다.

**Alternate Flow — 등록된 브랜드가 하나도 없음** (4 → 4a)
- 4a. 시스템은 빈 목록과 페이지 메타로 응답한다.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 페이지 번호가 음수이거나 페이지 크기가 허용 범위 밖 → `BAD_REQUEST` (400), 상태 변경 없음.

#### UC-3. (관리자) 브랜드 상세 조회

**Main Flow**
1. 관리자가 브랜드 식별자로 상세를 요청한다.
2. 시스템은 해당 브랜드의 존재를 확인한다 — 삭제 마크된 브랜드는 존재하지 않는 것으로 다룬다.
3. 시스템은 브랜드 정보를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 브랜드 식별자가 존재하지 않거나 삭제 마크되어 있음 → `BRAND_NOT_FOUND` (404), 상태 변경 없음.

#### UC-4. (관리자) 브랜드 등록

**Main Flow**
1. 관리자가 이름과 부가 정보(설명 등) 를 담아 브랜드 등록을 요청한다.
2. 시스템은 입력 필드의 필수·형식 규칙을 검증한다.
3. 시스템은 이름이 기존 브랜드(삭제 마크되지 않은) 와 중복되지 않는지 확인한다.
4. 시스템은 브랜드를 새로 등록한다.
5. 시스템은 등록된 브랜드 식별자와 정보를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 입력이 필수·형식 규칙을 어김 → `BRAND_BAD_REQUEST` (400), 등록 없음.
- **E2** (3): 이름이 기존 브랜드와 중복 → `DUPLICATE_BRAND_NAME` (409), 등록 없음.

#### UC-5. (관리자) 브랜드 정보 수정

**Main Flow**
1. 관리자가 브랜드 식별자와 변경할 정보(이름·설명 등) 를 담아 수정을 요청한다.
2. 시스템은 해당 브랜드의 존재를 확인한다 — 삭제 마크된 브랜드는 존재하지 않는 것으로 다룬다.
3. 시스템은 변경 입력의 필수·형식 규칙을 검증한다.
4. 이름이 변경되는 경우 시스템은 이름이 기존 브랜드와 중복되지 않는지 확인한다 — 자기 자신은 중복 대상에서 제외된다.
5. 시스템은 브랜드 정보를 갱신한다.
6. 시스템은 갱신된 브랜드 정보를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 브랜드 식별자가 존재하지 않거나 삭제 마크되어 있음 → `BRAND_NOT_FOUND` (404), 변경 없음.
- **E2** (3): 변경 입력이 필수·형식 규칙을 어김 → `BRAND_BAD_REQUEST` (400), 변경 없음.
- **E3** (4): 이름이 기존 브랜드와 중복 → `DUPLICATE_BRAND_NAME` (409), 변경 없음.

#### UC-6. (관리자) 브랜드 삭제 — 카스케이드

**Main Flow**
1. 관리자가 브랜드 식별자로 삭제를 요청한다.
2. 시스템은 해당 브랜드의 존재를 확인한다 — 삭제 마크된 브랜드는 존재하지 않는 것으로 다룬다.
3. 시스템은 단일 트랜잭션 안에서 다음을 모두 수행한다 — (a) 브랜드를 삭제 마크 처리한다, (b) 해당 브랜드 소속 상품들을 모두 삭제 마크 처리한다.
4. 시스템은 빈 페이로드(`data: null`) 와 성공 응답을 돌려준다.

**Alternate Flow — 브랜드에 소속 상품이 없음** (3 → 3a)
- 3a. 시스템은 브랜드만 삭제 마크 처리한다.

**Exception Flow**
- E0 에 위임 (관리자 인증 실패).
- **E1** (2): 브랜드 식별자가 존재하지 않거나 이미 삭제 마크되어 있음 → `BRAND_NOT_FOUND` (404), 상태 변경 없음.
- **E2** (3): 트랜잭션 도중 일부만 처리되는 상태가 발생하지 않는다 — 부분 실패 시 모든 변경이 함께 되돌려진다(롤백, 상태 변경 없음).

## 5. 도메인 규칙 / 불변식

**유일성**
- 브랜드 이름은 삭제 마크되지 않은 브랜드 사이에서 전체 유일하다.

**삭제 마크(Soft Delete)**
- 본 도메인의 모든 삭제는 행 제거가 아니라 삭제 마크다 — 행 자체는 보존된다.
- 삭제 마크된 브랜드는 회원·관리자 조회 어디에서도 노출되지 않는다 — 존재 확인에서는 "찾을 수 없음" 으로 다룬다.
- 브랜드 삭제는 그 브랜드 소속 모든 상품의 삭제 마크를 동반한다 — 두 변경은 같은 트랜잭션 경계 안에서 일어난다(부분 적용 없음). 상품 측 영향은 [product/requirements.md](../product/requirements.md) §5 참조.

**인증 채널 분리**
- 단일 브랜드 조회(UC-1) 는 인증을 필수로 하지 않는다.
- 관리자 동작(UC-2 ~ UC-6) 은 관리자 인증을 통과한 주체에게만 허용된다.
- 관리자 채널과 회원 채널은 노출 경로(`api-admin` vs `api`) 와 인증 방식이 분리된다.

> 데이터 구조 결정은 `logical-model.md` 를 참고한다 — 브랜드 전용 logical-model 은 아직 작성되지 않았으며, 관리자 인증의 데이터/메커니즘 결정은 [admin/logical-model.md](../admin/logical-model.md) 가 다룬다.

---

## 6. 비기능 요구사항

- 동일 이름의 브랜드 등록이 동시에 발생해도 유일성 약속을 위반하지 않는다.
- 모든 응답은 표준 응답 봉투(`ApiResponse<T>`) 로 감싼다.
