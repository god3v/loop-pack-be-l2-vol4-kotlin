## 1. 유비쿼터스 언어

| 용어 | 의미 |
|---|---|
| **사용자(User) / 회원** | 본 도메인이 다루는 사람. 영문 표기는 `User`, 한국어 호칭은 "회원" 으로 통일한다. |
| **사용자(미가입)** | 아직 회원으로 등록되지 않은 외부 행위자. 회원가입의 주체. |
| **LoginId** | 회원이 자신을 식별하기 위해 선택한 문자열. 영문/숫자만 허용되며 전체에서 유일하다. |
| **Password** | 회원이 본인 확인에 사용하는 비밀 문자열. 비밀번호 RULE 을 항상 만족한다. |
| **Email** | 회원의 이메일 주소. `local@domain.tld` 구조를 만족하며 전체에서 유일하다. |
| **비밀번호 RULE** | 길이(8~16) · 허용 문자집합(영문/숫자/특수) · 3카테고리 각 1자 이상 · 생년월일 미포함 의 네 조건. 가입과 변경에 동일하게 적용된다. |
| **헤더 인증** | 요청에 포함된 `X-Loopers-LoginId` / `X-Loopers-LoginPw` 두 헤더로 회원을 식별·확인하는 절차. |
| **이름 마스킹** | 이름의 마지막 1글자를 `*` 하나로 치환해 노출하는 규칙. |

---

## 2. 유저 스토리

- 사용자(미가입) 는 회원 가입을 할 수 있다.
- 회원은 자신의 정보를 조회할 수 있다 — 이름은 마지막 글자가 가려져 보인다.
- 회원은 자신의 비밀번호를 변경할 수 있다 — 본인 확인이 이중으로 이루어진다.

> 다음은 본 주차 범위 밖이다: 세션/JWT 인증, 소셜 로그인, 비밀번호 재설정 / 이메일 인증, 회원 정보 수정(이름·이메일·생년월일), 탈퇴, 관리자용 목록·검색.

---

## 3. 기능 요구사항

- 사용자(미가입) 가 가입 폼을 제출하면, 형식과 유일성을 모두 통과한 경우에만 회원으로 등록되고 성공 응답을 받는다.
- 가입 시 입력이 형식 규칙을 어기면 어떤 종류의 위반인지 카테고리로 구분된 실패 응답을 받는다 (일반 형식 위반과 비밀번호 RULE 위반이 구분된다).
- 가입 시 이미 존재하는 LoginId 또는 Email 이면 어떤 자원이 중복인지 구분된 실패 응답을 받는다.
- 회원이 인증 헤더와 함께 자신의 정보 조회를 요청하면 이름 마스킹이 적용된 본인 정보를 돌려받는다.
- 회원이 비밀번호 변경을 요청하면, 본인 확인이 이중으로 통과되고 새 비밀번호가 RULE 을 만족하며 현재 비밀번호와 달라야만 변경된다.
- 헤더 누락 / 계정 없음 / 비밀번호 불일치 / 변경 요청의 본문 prevPw 불일치는 회원 입장에서 동일하게 "인증 실패" 로 보인다 — 어느 원인인지 응답에서 드러나지 않는다.
- 회원의 비밀번호는 어떤 응답에서도 노출되지 않는다 — 평문도, 해시도 노출되지 않는다.

## 4. 유스케이스 흐름

### 공통 사전 단계 — 헤더 인증

UC-2, UC-3 의 Main Flow 진입 전에 항상 선행한다.

1. 시스템은 `X-Loopers-LoginId` 와 `X-Loopers-LoginPw` 두 헤더의 존재를 확인한다.
2. 시스템은 LoginId 로 회원을 조회하고, 저장된 비밀번호와 헤더 값의 일치를 확인한다.
3. 두 검사가 모두 통과하면 해당 회원을 컨텍스트로 두고 UC 본 흐름으로 진입한다.

**Exception**
- **E0** (사전 단계 어디서든): 헤더 누락 / 회원 없음 / 비밀번호 불일치 중 하나라도 발생 → `UNAUTHORIZED` (401), 상태 변경 없음. 어느 원인인지 응답에서 구분하지 않는다.

### UC-1. 회원 가입

**Main Flow**
1. 사용자(미가입) 가 LoginId · Password · 이름 · 생년월일 · Email 을 제출한다.
2. 시스템은 각 필드의 형식·길이 규칙을 검증한다.
3. 시스템은 Password 가 비밀번호 RULE 을 만족하는지 검증한다.
4. 시스템은 LoginId 와 Email 의 유일성을 확인한다.
5. 시스템은 회원을 등록하고 `200 OK` 와 빈 페이로드(`data: null`) 를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- **E1** (2): LoginId / 이름 / 생년월일 / Email 형식·길이 위반 → `SIGNUP_BAD_REQUEST` (400), 등록 없음.
- **E2** (3): 비밀번호 RULE 위반 → `INVALID_PASSWORD` (400), 등록 없음.
- **E3** (4): LoginId 중복 → `DUPLICATE_LOGIN_ID` (409), 등록 없음.
- **E4** (4): Email 중복 → `DUPLICATE_EMAIL` (409), 등록 없음.

### UC-2. 내 정보 조회

> 공통 사전 단계(헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 자신의 정보 조회를 요청한다.
2. 시스템은 LoginId · 이름(마스킹 적용) · 생년월일 · Email 을 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).

### UC-3. 비밀번호 수정

> 공통 사전 단계(헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 `prevPw` 와 `nextPw` 를 담아 변경을 요청한다.
2. 시스템은 헤더의 비밀번호와 본문의 `prevPw` 가 일치하는지 한 번 더 확인한다 (이중 본인 확인).
3. 시스템은 `nextPw` 가 비밀번호 RULE 을 만족하는지 검증한다.
4. 시스템은 `nextPw` 가 현재 비밀번호와 다른지 확인한다.
5. 시스템은 회원의 비밀번호를 갱신하고 `200 OK` 와 빈 페이로드를 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (2): 본문 `prevPw` 가 헤더 비밀번호와 불일치 → `UNAUTHORIZED` (401), 변경 없음. (E0 와 동일한 응답 — 원인 구분하지 않음)
- **E2** (3): `nextPw` 가 비밀번호 RULE 위반 → `INVALID_PASSWORD` (400), 변경 없음.
- **E3** (4): `nextPw` 가 현재 비밀번호와 동일 → `PASSWORD_CHANGE_BAD_REQUEST` (400), 변경 없음.

## 5. 도메인 규칙 / 불변식

- `LoginId` · `Password` · `Email` 은 값 객체다. 형식 검증은 생성 시점에 수행되며, 인스턴스는 항상 유효한 상태다.
- 비밀번호 RULE 위반은 가입과 비밀번호 변경에서 같은 의미로 다뤄진다 — 두 맥락에서 동일한 실패 의미를 가진다.
- "새 비밀번호 ≠ 현재 비밀번호" 는 값 객체가 아닌 회원의 **변경 행위** 에 속하는 규칙이다.
- 헤더 인증(LoginId 조회 + Password 일치 확인) 의 결합은 응용 계층의 책임이며, 도메인은 형식 검증과 동등성 비교까지만 책임진다.
- 헤더 인증 실패의 원인(헤더 누락 / 계정 없음 / 비밀번호 불일치 / 본문 prevPw 불일치) 은 응답에서 구분하지 않는다 — 계정 존재 여부를 외부에 노출하지 않기 위함이다.
- 회원의 비밀번호는 어떤 외부 응답 DTO 와 로그에도 포함되지 않는다 — 해시 형태조차 포함하지 않는다.
- 이름은 가입 시점에 최소 2글자가 보장되므로, 마스킹 시 1글자 경계 케이스는 발생하지 않는다.
- 비밀번호는 **장기적으로** 단방향 해시로 저장한다. 현재 구현은 평문 비교다 — **TBD**: BCrypt 도입 시점과 마이그레이션 전략.

> 데이터 구조 결정은 [docs/domain/user/logical-model.md](./logical-model.md) 를 참고한다.

## 6. 비기능 요구사항

- 모든 응답은 표준 응답 봉투(공통 메타 + data) 로 감싼다. No Content 성공도 `data: null` 형태를 유지한다.
- 도메인 예외는 표준 변환을 거쳐 일관된 실패 응답 형식으로 노출된다.
- 비밀번호 평문/해시는 어떤 형태로도 로그에 출력하지 않는다.
- 쓰기 작업은 트랜잭션 경계 안에서 수행하고, 읽기 작업은 읽기 전용 모드로 수행한다.
- 시간대는 `Asia/Seoul` 기준이며, 만 나이 계산은 가입 시점(`now()`) 기준이다.

# 미해결 결정 사항

- **BCrypt 해시 저장 도입** — 현재 `Password` 의 일치 비교는 평문 동등성 비교다. 어떤 시점에 어떤 알고리즘(BCrypt / Argon2 등) 으로 전환할지, 기존 평문 저장본의 마이그레이션 전략(다음 로그인 시 재해시 / 일괄 변환 / 강제 리셋) 을 어떻게 잡을지 결정되지 않았다.
