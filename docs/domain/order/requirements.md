# 주문(Order) 도메인 요구사항 명세

> **상태**: **v0.5 초안**
> **작성일**: 2026-06-10
> **변경 이력**:
> - v0.5 (2026-06-11): **결제 실패 보상·`PaymentFacade` 추출 반영(구현 동기화).** (1) 커밋 후 결제 오케스트레이션이 `application.order.PaymentFacade.pay(orderId)`(@Transactional REQUIRES_NEW) 로 이동했고, `OrderPaymentEventListener` 는 `AFTER_COMMIT` 에 Facade 를 위임하는 얇은 트리거가 되었다. (2) 결제 실패 시 같은 트랜잭션에서 재고 원복(`Product.restoreStock`)·쿠폰 사용 취소(`UserCoupon.cancelUse`) 보상 후 `PAYMENT_FAILED` 로 전이하는 흐름을 구현·테스트로 편입(멱등 가드 포함) — `PAYMENT_FAILED` 는 이제 도달 가능한 종료 상태다. (3) "결제 항상 성공 → 실패·보상 미발생/차주 과제" 단정을 폐기하고, 운영 게이트웨이가 항상 성공하는 mock(`AlwaysSuccessPaymentGateway`) 이라 정상 실행에서만 실패가 안 나는 것으로 정정. "**실제 외부 PG 연동(mock 교체)**" 만 미해결로 남긴다.
> - v0.4 (2026-06-10): **쿠폰 연동 방식·식별자 명명·결제 게이트웨이 재결정(설계 리뷰 반영).** (1) 쿠폰 사용 조율 주체를 **주문 도메인으로 명시** — 주문이 발급 쿠폰을 직접 조율(소유·사용·만료·최소금액 검증 + 소진)하고, 할인 계산·단일 사용·만료 판정 등 쿠폰 **규칙** 만 Coupon 도메인 객체에 위임한다. (v0.3 의 "사용 검증·소진은 Coupon 도메인 책임" 문구를 폐기.) (2) 주문이 참조하는 쿠폰 식별자를 **발급 쿠폰 식별자(userCouponId)** 로 전면 명명한다 — 쿠폰 템플릿(couponId) 과 구분. (3) **외부 결제 게이트웨이를 본 iteration 범위로 편입** — 단, 항상 성공하는 최소 어댑터로 두고, **주문 저장 트랜잭션이 커밋된 뒤** 호출한다(외부 호출을 DB 트랜잭션에 가두지 않는다). 주문 흐름은 `재고·쿠폰·주문(PAYMENT_PENDING) 단일 트랜잭션 → 커밋 → 결제 → PAID 전이` 2단계가 된다.
> - v0.3 (2026-06-09): 쿠폰 사용 흡수 + 원자성 모델 단순화. 쿠폰 사용 흐름(정액/정률·단일 사용·3금액 스냅샷) 편입, 단일 트랜잭션 전체 롤백 모델 도입, 외부 결제·`PAYMENT_FAILED`·보상을 차주 이관. (v0.4 에서 결제·쿠폰 조율 주체·명명을 재결정.)
> - v0.2 (2026-05-20): 어드민 시각 흡수. 운영자(Admin) · LDAP 인증 · 운영 메타 · 회원 표시명 · 어드민 API 경로 어휘를 §1 에 추가하고, 어드민 주문 목록 조회·단일 상세 조회를 UC-4·UC-5 로 추가한다.
> - v0.1 (2026-05-20): 최초 초안. 주문 생성·결제 흐름, 멱등 키 기반 중복 흡수, 기간·페이지 단위 목록 조회, 단일 상세의 권한 응답 정책을 정의한다.

## 1. 유비쿼터스 언어

| 용어 | 의미 |
|---|---|
| **주문(Order)** | 회원의 구매 의사를 표현하는 거래 단위. 영문 표기는 `Order`. 정확히 한 명의 회원에 소속되고, 1개 이상의 주문 라인을 가진다. 주문 시각·금액 스냅샷·상태·적용 발급 쿠폰 식별자·결제 메타를 가진다. |
| **주문 라인(OrderLine)** | 한 주문에 속한 한 상품-수량 단위. 상품 식별자·수량과 함께 **시점 가격(`unitPrice`)** · **시점 상품명(`productName`)** 의 스냅샷을 박아 보관한다. |
| **주문 상태(OrderStatus)** | 주문 생명주기의 상태. 주문 저장 트랜잭션 직후의 상태는 `결제대기(PAYMENT_PENDING)` 이고, 커밋 후 결제 결과에 따라 `결제완료(PAID)` 또는 `결제실패(PAYMENT_FAILED)` 로 전이한다. `PAYMENT_FAILED` 는 결제 실패 시 도달하는 종료 상태이며, 그 전이에는 재고 원복·쿠폰 사용 취소 보상이 함께 따른다 — 이후 조회(GET)에서 그 상태로 보인다. 단, 운영용 결제 게이트웨이는 여전히 항상 성공하는 mock(`AlwaysSuccessPaymentGateway`) 이라 정상 실행에서는 `PAYMENT_FAILED` 가 나타나지 않는다(실패 경로·보상은 테스트로 커버). |
| **상품 합계(originalAmount)** | 주문 라인의 `unitPrice × 수량` 을 모두 합산한 금액. **쿠폰 적용 전 금액** 이다. |
| **할인 금액(discountAmount)** | 적용된 쿠폰으로 상품 합계에서 차감되는 금액. 쿠폰 미적용 시 `0` 이다. 음수가 될 수 없고, 상품 합계를 초과하지 않는다. |
| **결제 금액(totalAmount)** | `상품 합계 − 할인 금액` 으로 산출되는 **최종 결제 대상 금액**. 0 이상이다. |
| **회원(User)** | 회원 시각 유스케이스의 주체이자 모든 주문의 데이터 주체. User 도메인이 정의한 헤더 인증 절차를 그대로 사용한다. 비회원은 회원 시각의 어떤 흐름에도 진입할 수 없다. |
| **상품(Product)** | 주문 라인의 대상. Product 도메인이 정의하며, 본 도메인은 그 존재 여부·시점 가격·이름을 외부 의존으로 사용하고 재고 차감 호출의 발기인이 된다. |
| **재고(Stock)** | 상품의 잔여 수량. Product 도메인의 개념이며, 주문 합의 시 본 도메인이 차감을 조율한다. 차감 실패는 주문 실패로 직결된다. |
| **쿠폰(Coupon) / 발급 쿠폰(UserCoupon)** | 쿠폰(Coupon)은 정액(`FIXED`)·정률(`RATE`) 할인 정책 템플릿, 발급 쿠폰(UserCoupon)은 회원이 보유한 1회용 인스턴스다. Coupon 도메인이 정의한다([coupon/requirements.md](../coupon/requirements.md)). **주문이 참조하는 식별자는 발급 쿠폰 식별자(`userCouponId`)** 다 — 템플릿 식별자(`couponId`) 가 아니다. |
| **쿠폰 적용(Coupon Application)** | 주문 생성 시 회원이 보유한 사용 가능한 발급 쿠폰 1장을 주문에 적용해, 할인 금액을 상품 합계에 반영하고 그 발급 쿠폰을 즉시 사용 완료(소진) 시키는 행위. **본 도메인이 이 적용을 직접 조율한다** — 발급 쿠폰을 동시성 안전하게 조회해 소유·사용 여부·만료·최소 주문 금액을 검증하고, 할인 계산과 사용 완료 전이를 Coupon 도메인 객체에 위임한다. |
| **외부 결제 게이트웨이(Payment Gateway)** | 결제 처리를 위임하는 외부 의존. 본 도메인의 outbound 게이트웨이로, 결제 금액을 받아 결제 결과(트랜잭션 식별자·결과 코드·성공 여부) 를 돌려준다. **본 iteration 에서는 항상 성공하는 최소 구현** 을 사용하며, 주문 저장 트랜잭션이 **커밋된 뒤** 호출된다. |
| **결제 메타(Payment Metadata)** | 결제 결과로 주문에 박히는 정보 — 결제 트랜잭션 식별자, 결제 결과 코드. 운영 메타의 일부로 어드민 응답에 노출된다. |
| **헤더 인증** | User 도메인이 정의한 `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 기반 식별 절차. 회원 시각의 모든 유스케이스 진입 전 선행한다. |
| **멱등 키(Idempotency Key)** | 주문 생성 요청의 중복 흡수를 위해 클라이언트가 발급하는 식별 문자열. 같은 회원의 같은 키 재요청은 기존 주문과 동일한 응답으로 수렴한다. |
| **페이지(Page)** | 주문 목록의 분할 단위. `page` (0부터) 와 `size` 두 값으로 식별되는 offset 페이징을 사용한다. 미지정 시 `page=0` / `size=20` 이 적용된다. |
| **운영자(Admin)** | 회사 내부에서 LDAP 으로 식별된 사람. CS·정산·운영을 위해 본 도메인에 접근하는 액터이며, 어드민 시각 유스케이스의 주체다. 회원이 아니다. |
| **LDAP 인증** | 운영자의 식별·인가를 사내 LDAP 으로 처리하는 절차. 본 명세는 그 절차를 추상 의존으로만 둔다. |
| **운영 메타(Admin Metadata)** | 회원 응답에는 노출되지 않지만 운영자 응답에는 추가로 노출되는 정보 묶음 — 회원 식별자, 회원 표시명, 결제 트랜잭션 식별자, 결제 결과 코드. |
| **회원 표시명** | 운영자 응답에 회원을 식별 가능하게 보여주기 위한 문자열. User 도메인의 이름 마스킹 규칙(마지막 1글자를 `*` 로 치환) 을 그대로 적용한다. |
| **어드민 API 경로** | 회원 API 와 분리된 게이트로, `/api-admin/v1/...` 로 노출된다. |

## 2. 유저 스토리

**회원 시각**
- 회원은 1개 이상의 상품-수량(라인 아이템) 을 담아 주문을 요청할 수 있다.
- 회원은 주문 시 자신이 보유한 사용 가능한 발급 쿠폰 1장을 적용해 할인받을 수 있다.
- 회원은 쿠폰을 적용한 주문에서 상품 합계·할인 금액·결제 금액을 모두 확인할 수 있다.
- 회원은 적용할 수 없는 쿠폰(없음 / 이미 사용됨 / 만료됨 / 타인 소유 / 최소 주문 금액 미달) 으로 요청하면 주문이 실패함을 기대할 수 있다.
- 회원은 재고 차감·쿠폰 적용 중 어느 한 단계라도 실패하면 어떤 변경(재고·쿠폰·주문)도 남지 않음을 기대할 수 있다.
- 회원은 같은 멱등 키로 주문 요청을 여러 번 보내도 한 주문만 만들어짐을 기대할 수 있다.
- 회원은 자신의 주문 목록을 기간으로 조회할 수 있다.
- 회원은 자신의 단일 주문 상세를 조회할 수 있다 — 주문 시점의 가격과 상품명이 그대로 보인다.
- 회원은 자기 자신의 주문만 조회할 수 있다 — 타인의 주문은 단일 조회로도 보지 못한다.

**운영자 시각**
- 운영자는 전사의 주문 목록을 페이지 단위로 조회할 수 있다.
- 운영자는 단일 주문 식별자로 그 주문의 상세를 조회할 수 있다 — 어느 회원의 주문이든 본인 한정 제약 없이 본다.
- 운영자는 주문 응답에서 회원 식별자와 운영 메타(결제 트랜잭션 식별자·결과 코드 등) 를 함께 본다.
- 운영자는 카드 번호 · CVC 등 결제 민감 정보를 어떤 응답에서도 보지 않는다.

> 본 명세 범위 밖이다: 주문 취소, 포인트 차감/적립, 비회원 주문, **실제 외부 결제 게이트웨이 연동(mock 교체)** — 결제 실패·보상·`PAYMENT_FAILED` 전이 자체는 구현·테스트되었고, 다만 운영 어댑터를 항상 성공하는 mock(`AlwaysSuccessPaymentGateway`) 에서 실제 PG 로 교체하는 일만 향후 과제다 — 쿠폰 템플릿의 발급·관리(Coupon 도메인 소유).

## 3. 기능 요구사항

사용자 관점에서 관찰 가능한 동작만 적는다.

**회원 시각**
- 회원이 인증 헤더와 멱등 키, 1개 이상의 라인 아이템(상품 식별자 · 수량), 선택적 발급 쿠폰 식별자를 담아 주문을 요청하면, 재고 차감과 (쿠폰을 동반한 경우) 쿠폰 적용과 결제가 모두 성공한 경우에만 주문이 `결제완료` 로 확정되고 상품 합계 · 할인 금액 · 결제 금액이 함께 응답된다.
- 회원이 쿠폰 없이 주문을 요청하면 할인 금액은 `0` 이고 결제 금액은 상품 합계와 같다.
- 회원이 사용 가능한 자기 소유 쿠폰을 적용해 주문에 성공하면, 할인 금액이 결제 금액에 반영되고 그 쿠폰은 다시 사용할 수 없게 된다.
- 회원이 존재하지 않거나 타인 소유인 발급 쿠폰 식별자로 주문을 요청하면 실패 응답을 받는다 — 주문이 만들어지지 않고 재고도 차감되지 않는다.
- 회원이 이미 사용한 쿠폰으로 주문을 요청하면 실패 응답을 받는다 — 주문이 만들어지지 않는다.
- 회원이 만료된 쿠폰으로 주문을 요청하면 실패 응답을 받는다 — 주문이 만들어지지 않는다.
- 회원이 쿠폰의 최소 주문 금액에 못 미치는 주문에 그 쿠폰을 적용하면 실패 응답을 받는다 — 주문이 만들어지지 않는다.
- 회원이 라인 아이템에 존재하지 않는 상품 식별자를 담아 보내면 실패 응답을 받는다 — 어떤 라인도 처리되지 않고 주문이 만들어지지 않는다.
- 회원의 라인 아이템 중 하나라도 재고가 부족하면 실패 응답을 받는다 — 어떤 재고 차감도, 어떤 쿠폰 소진도 남지 않는다.
- 회원의 주문 저장 단계(재고·쿠폰·주문)에서 어느 하나라도 실패하면 그 시점까지의 모든 변경이 함께 취소된다 — 부분 반영은 존재하지 않는다.
- 회원이 같은 멱등 키로 주문을 다시 요청하면 신규 주문이 만들어지지 않고 기존 주문과 동일한 응답을 받는다.
- 회원이 자신의 주문 목록을 `startAt` · `endAt` · 페이지 번호 · 페이지 크기와 함께 요청하면, 기간 안의 자신의 주문이 주문 시각 내림차순으로 페이지 단위로 응답된다.
- 회원이 페이지 번호 또는 페이지 크기를 허용 범위 밖으로 보내면 실패 응답을 받는다.
- 회원이 페이지 번호 · 페이지 크기를 지정하지 않으면 기본값(`page=0` / `size=20`) 으로 응답된다.
- 회원이 단일 주문 식별자로 상세를 요청하면 주문 정보(주문 시각 · 상태 · 상품 합계 · 할인 금액 · 결제 금액 · 적용 발급 쿠폰 식별자) 와 라인 아이템(시점 가격 · 시점 상품명 · 수량) 을 응답받는다.
- 회원이 타인의 주문 식별자로 단일 조회를 요청하면 권한 실패 응답을 받는다 — 존재하지 않는 식별자와 응답이 구분된다.
- 회원이 존재하지 않는 주문 식별자로 단일 조회를 요청하면 "찾을 수 없음" 실패 응답을 받는다.
- 인증되지 않은 회원 요청은 회원 시각의 어느 흐름에도 진입할 수 없다.

**운영자 시각**
- 운영자가 LDAP 인증과 페이지 번호 · 페이지 크기를 담아 주문 목록을 요청하면, 전사 주문이 주문 시각 내림차순으로 페이지 단위로 응답된다.
- 운영자가 페이지 번호 · 페이지 크기를 지정하지 않으면 기본값(`page=0` / `size=20`) 으로 응답된다.
- 운영자가 페이지 번호 또는 페이지 크기를 허용 범위 밖으로 보내면 실패 응답을 받는다.
- 운영자가 단일 주문 식별자로 상세를 요청하면 주문 정보(상품 합계 · 할인 금액 · 결제 금액 포함) 와 라인 아이템과 운영 메타를 함께 응답받는다.
- 운영자가 존재하지 않는 주문 식별자로 단일 조회를 요청하면 "찾을 수 없음" 실패 응답을 받는다.
- 운영자는 본인 한정 제약 없이 어느 회원의 주문이든 조회할 수 있다.
- LDAP 인증에 실패한 요청은 어드민 시각의 어느 흐름에도 진입할 수 없다.
- LDAP 으로 식별은 됐으나 주문 조회 권한이 없는 운영자의 요청은 권한 실패 응답을 받는다.
- 회원의 카드 번호 · CVC 등 결제 민감 정보는 운영 메타에도 포함되지 않는다.

## 4. 유스케이스 흐름

### 공통 사전 단계 A — 헤더 인증 (회원 시각)

UC-1, UC-2, UC-3 의 Main Flow 진입 전에 항상 선행한다.

1. 시스템은 `X-Loopers-LoginId` 와 `X-Loopers-LoginPw` 두 헤더의 존재를 확인한다.
2. 시스템은 LoginId 로 회원을 조회하고, 저장된 비밀번호와 헤더 값의 일치를 확인한다.
3. 두 검사가 모두 통과하면 해당 회원을 컨텍스트로 두고 UC 본 흐름으로 진입한다.

**Exception**
- **E0** (사전 단계 어디서든): 헤더 누락 / 회원 없음 / 비밀번호 불일치 중 하나라도 발생 → `UNAUTHORIZED` (401), 상태 변경 없음.

### 공통 사전 단계 B — LDAP 인증 (어드민 시각)

UC-4, UC-5 의 Main Flow 진입 전에 항상 선행한다.

**Exception**
- **E0-a** (사전 단계): LDAP 식별 실패 → `UNAUTHORIZED` (401), 상태 변경 없음.
- **E0-b** (사전 단계): LDAP 식별은 통과했으나 주문 조회 권한 부족 → `FORBIDDEN` (403), 상태 변경 없음.

### UC-1. 주문 생성 (회원)

> 공통 사전 단계 A (헤더 인증) 를 먼저 통과한다.
> 본 유스케이스는 **2단계** 다. ① 재고 차감 · 쿠폰 사용 · 주문 저장(`결제대기`) 을 **하나의 트랜잭션** 으로 처리한다 — 어느 한 단계라도 실패하면 그 시점까지의 모든 변경이 함께 취소되고 주문은 생성되지 않는다(전부 반영 또는 전부 롤백). 재고 차감과 쿠폰 사용의 상대 순서는 결과에 영향을 주지 않는다. ② 그 트랜잭션이 **커밋된 뒤** 외부 결제 게이트웨이에 결제를 요청하고, 결과를 반영해 주문을 `결제완료(PAID)` 또는 실패 시 보상 후 `결제실패(PAYMENT_FAILED)` 로 전이시킨다 — 결제 호출은 DB 트랜잭션 안에 두지 않는다. 이 ② 단계의 조율(외부 결제 호출 → 결과 반영 → 실패 시 보상)·트랜잭션 경계는 `application.order.PaymentFacade.pay(orderId)` 가 단일 소유하며, 이벤트 리스너는 커밋 후 그 Facade 를 호출하는 얇은 트리거다.

**Main Flow**
1. 회원이 멱등 키, 1개 이상의 라인 아이템(상품 식별자 · 수량), 선택적 발급 쿠폰 식별자(`userCouponId`) 를 담아 주문 생성을 요청한다.
2. 시스템은 인증된 회원과 멱등 키의 조합으로 기존 주문이 있는지 확인한다.
3. (트랜잭션 시작) 시스템은 각 라인 아이템의 상품 존재를 확인하고, 시점 가격과 시점 상품명을 스냅샷한다.
4. 시스템은 상품 합계(라인 `unitPrice × 수량` 의 합) 를 산출한다.
5. 시스템은 각 라인 아이템의 재고를 차감한다.
6. 시스템은 발급 쿠폰 식별자가 있으면, 그 쿠폰을 동시성 안전하게 조회해 인증된 회원의 소유이며 아직 사용되지 않았고 만료되지 않았으며 상품 합계가 최소 주문 금액 이상인지 검증한 뒤, 할인 금액을 계산하고 그 쿠폰을 사용 완료로 전이시킨다. 쿠폰이 없으면 할인 금액은 `0` 이다.
7. 시스템은 결제 금액(`상품 합계 − 할인 금액`) 을 확정해 주문을 `결제대기(PAYMENT_PENDING)` 로 저장하고 **주문 발생 이벤트를 발행**한 뒤, `200 OK` 와 함께 주문 식별자 · 주문 시각 · 상태(`PAYMENT_PENDING`) · 상품 합계 · 할인 금액 · 결제 금액 · 적용 발급 쿠폰 식별자 · 라인 아이템 스냅샷을 응답한다. (여기서 트랜잭션이 커밋된다.)
8. **트랜잭션 커밋 후**, 이벤트를 받은 처리기(`OrderPaymentEventListener`, `AFTER_COMMIT`) 가 `PaymentFacade.pay(orderId)` 에 위임한다. `pay` 는 **별도 트랜잭션**(`REQUIRES_NEW` — `AFTER_COMMIT` 콜백은 이미 커밋된 원 트랜잭션이 바인딩돼 있어 평범한 `@Transactional` 은 조인만 하고 커밋되지 않기 때문) 에서 주문을 다시 로드하고, 외부 결제 게이트웨이에 결제 금액의 결제를 요청한다. 진입 시 주문 상태가 `결제대기` 가 아니면 즉시 종료한다(멱등 가드 — 중복 트리거 시 이중 청구·이중 보상 방지).
9. 처리기는 결제 결과(트랜잭션 식별자 · 결과 코드) 를 반영해 같은 트랜잭션에서 주문 상태를 전이·저장한다.
   - **성공** → 주문을 `결제완료(PAID)` 로 전이한다. 이후 조회(UC-2·3·4·5) 는 `PAID` 상태와 결제 메타를 보여준다.
   - **실패** (9a 참조) → 같은 트랜잭션에서 보상 후 주문을 `결제실패(PAYMENT_FAILED)` 로 전이한다.

**Alternate Flow — 결제 실패 시 보상** (9 → 9a)
- 9a. 결제가 실패하면 `PaymentFacade` 는 같은 `REQUIRES_NEW` 트랜잭션에서 1단계의 변경을 보상한다 — (1) 주문 라인별로 차감했던 재고를 `Product.restoreStock(quantity)` 로 원복하고, (2) 적용된 발급 쿠폰이 있으면 `UserCoupon.cancelUse()` 로 사용을 취소(USED→AVAILABLE, `usedAt=null`; 사용 전이면 멱등 no-op)한 뒤, (3) `Order.markPaymentFailed(...)` 로 주문을 `결제실패(PAYMENT_FAILED)` 로 전이·저장한다. 운영용 결제 게이트웨이는 항상 성공하는 mock 이라 정상 실행에서는 이 분기가 발생하지 않으며, 실패 경로·보상은 테스트(`OrderPaymentFailureIntegrationTest` 통합 + `PaymentFacadeTest` 단위) 로 커버된다.

**Alternate Flow — 쿠폰 미적용** (6 → 6a)
- 6a. 발급 쿠폰 식별자가 없으면 할인 금액을 `0` 으로 두고 7 로 진행한다. 결제 금액은 상품 합계와 같다.

**Alternate Flow — 같은 멱등 키 재요청** (2 → 2a)
- 2a. 시스템은 새 주문을 만들지 않고, 기존 주문과 동일한 응답을 그대로 돌려준다 (멱등 수렴).

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (1): 멱등 키가 비어 있거나, 라인 아이템이 비어 있거나, 어떤 라인의 수량이 1 미만 → `BAD_REQUEST` (400), 상태 변경 없음.
- **E2** (3): 라인 아이템 중 하나라도 상품이 존재하지 않음 → `PRODUCT_NOT_FOUND` (404), 트랜잭션 전부 롤백, 주문 생성 없음.
- **E3** (5): 라인 아이템 중 하나라도 재고가 부족 → `INSUFFICIENT_STOCK` (409), 트랜잭션 전부 롤백(차감된 재고 없음), 주문 생성 없음.
- **E4** (6): 발급 쿠폰이 존재하지 않거나 인증된 회원의 소유가 아님 → `USER_COUPON_NOT_FOUND` (404), 트랜잭션 전부 롤백(차감된 재고 원복), 주문 생성 없음.
- **E5** (6): 발급 쿠폰이 이미 사용됨 → `ALREADY_USED_COUPON` (409), 트랜잭션 전부 롤백, 주문 생성 없음.
- **E6** (6): 쿠폰이 만료되었거나 상품 합계가 최소 주문 금액에 못 미침 → `COUPON_NOT_APPLICABLE` (400), 트랜잭션 전부 롤백, 주문 생성 없음.

> 8~9단계(결제) 의 실패 분기와 보상(재고 원복 · 쿠폰 사용 취소 · `PAYMENT_FAILED` 전이) 은 구현·테스트되어 있다(9a 참조). 다만 운영용 결제 게이트웨이가 항상 성공하는 mock(`AlwaysSuccessPaymentGateway`) 이라 정상 실행에서는 실패 분기가 발생하지 않는다. 실제 외부 PG 로 mock 을 교체하는 일만 향후 과제다(§미해결). 주문 생성 응답(7단계) 은 결제가 커밋 후 비동기로 일어나므로 항상 `결제대기(PAYMENT_PENDING)` 이고, 결제 실패는 이후 조회에서 `PAYMENT_FAILED`(+ 재고·쿠폰 원복) 로 드러난다 — 생성 엔드포인트에는 결제-실패 에러 응답이 없다.

### UC-2. 내 주문 목록 조회 (회원)

> 공통 사전 단계 A (헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 `startAt` (선택) · `endAt` (선택) · 페이지 번호(선택) · 페이지 크기(선택) 를 담아 주문 목록을 요청한다.
2. 시스템은 페이지 번호 · 페이지 크기가 허용 범위 안에 있는지, `startAt ≤ endAt` 인지 검증한다.
3. 시스템은 미지정 입력을 기본값으로 대체한다 — 페이지 번호는 `0`, 페이지 크기는 `20`.
4. 시스템은 인증된 회원의 주문 중 주문 시각이 기간 안에 든 것들을 주문 시각 내림차순으로 페이지 단위로 응답한다.
5. 응답 각 항목은 주문 식별자 · 주문 시각 · 상태 · 상품 합계 · 할인 금액 · 결제 금액 · 라인 아이템 요약을 포함한다.

**Alternate Flow — 결과가 비어 있음** (4 → 4a)
- 4a. 시스템은 빈 목록과 페이지 메타로 응답한다 (실패가 아니다).

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (2): 페이지 번호가 음수이거나 페이지 크기가 허용 범위를 벗어남 → `BAD_REQUEST` (400), 상태 변경 없음.
- **E2** (2): `startAt` 이 `endAt` 보다 큼 → `INVALID_DATE_RANGE` (400), 상태 변경 없음.

### UC-3. 단일 주문 상세 조회 (회원)

> 공통 사전 단계 A (헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 주문 식별자로 단일 주문 상세를 요청한다.
2. 시스템은 해당 주문의 존재를 확인한다.
3. 시스템은 그 주문의 소유 회원이 인증된 회원과 일치하는지 확인한다.
4. 시스템은 주문 정보(주문 식별자 · 주문 시각 · 상태 · 상품 합계 · 할인 금액 · 결제 금액 · 적용 발급 쿠폰 식별자) 와 라인 아이템(상품 식별자 · 시점 상품명 · 시점 가격 · 수량) 을 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (2): 주문 식별자가 존재하지 않음 → `ORDER_NOT_FOUND` (404), 상태 변경 없음.
- **E2** (3): 주문의 소유 회원이 인증된 회원과 다름 → `FORBIDDEN` (403), 상태 변경 없음. (E1 과 응답이 분리되어 "존재 여부" 와 "권한 없음" 의 의미가 구분된다.)

### UC-4. 어드민 주문 목록 조회 (운영자)

> 공통 사전 단계 B (LDAP 인증) 를 먼저 통과한다.

**Main Flow**
1. 운영자가 페이지 번호(선택) · 페이지 크기(선택) 를 담아 주문 목록을 요청한다.
2. 시스템은 페이지 번호 · 페이지 크기가 허용 범위 안에 있는지 검증한다.
3. 시스템은 미지정 입력을 기본값(`page=0` / `size=20`) 으로 대체한다.
4. 시스템은 전체 주문을 주문 시각 내림차순으로 페이지 단위로 응답한다.
5. 응답 각 항목은 주문 식별자 · 주문 시각 · 상태 · 상품 합계 · 할인 금액 · 결제 금액 · 라인 아이템 요약 · 운영 메타(회원 식별자, 회원 표시명, 결제 트랜잭션 식별자, 결제 결과 코드) 를 포함한다.

**Alternate Flow — 결과가 비어 있음** (4 → 4a)
- 4a. 시스템은 빈 목록과 페이지 메타로 응답한다 (실패가 아니다).

**Exception Flow**
- E0-a / E0-b 에 위임 (LDAP 인증·인가 실패).
- **E1** (2): 페이지 번호가 음수이거나 페이지 크기가 허용 범위를 벗어남 → `BAD_REQUEST` (400), 상태 변경 없음.

### UC-5. 어드민 단일 주문 상세 조회 (운영자)

> 공통 사전 단계 B (LDAP 인증) 를 먼저 통과한다.

**Main Flow**
1. 운영자가 주문 식별자로 단일 주문 상세를 요청한다.
2. 시스템은 해당 주문의 존재를 확인한다.
3. 시스템은 주문 정보(주문 식별자 · 주문 시각 · 상태 · 상품 합계 · 할인 금액 · 결제 금액) 와 라인 아이템과 운영 메타를 응답한다.

**Alternate Flow** — 없음. 운영자는 어느 회원의 주문이든 볼 수 있으므로 UC-3 의 "타인의 주문" 분기가 본 흐름에는 존재하지 않는다.

**Exception Flow**
- E0-a / E0-b 에 위임 (LDAP 인증·인가 실패).
- **E1** (2): 주문 식별자가 존재하지 않음 → `ORDER_NOT_FOUND` (404), 상태 변경 없음.

## 5. 도메인 규칙 / 불변식

**모델·금액 스냅샷**
- 주문은 정확히 한 명의 회원에 속한다 — 회원 없는 주문은 존재하지 않는다.
- 주문은 1개 이상의 라인 아이템을 가진다 — 빈 주문은 존재하지 않는다.
- 라인 아이템의 수량은 1 이상의 정수다.
- 라인 아이템은 시점 가격(`unitPrice`) 과 시점 상품명(`productName`) 의 스냅샷을 보관한다 — 이후 카탈로그 변경이 과거 주문의 금액이나 표시명을 바꾸지 않는다.
- 주문은 **3금액 스냅샷** 을 보관한다 — 상품 합계(쿠폰 적용 전), 할인 금액, 결제 금액. 세 값은 주문 시점에 확정되어 이후 변하지 않는다.
- 상품 합계는 라인 아이템의 `unitPrice × 수량` 합산이다.
- 할인 금액은 음수가 될 수 없고 상품 합계를 초과하지 않는다 — 쿠폰 미적용 시 `0` 이다.
- 결제 금액은 `상품 합계 − 할인 금액` 이며 0 이상이다.
- 주문은 적용된 발급 쿠폰의 식별자(`userCouponId`) 를 보관한다 — 쿠폰 미적용 시 비어 있다.

**상태 전이**
- 주문 상태는 `결제대기(PAYMENT_PENDING)` 에서 출발해, 결제 결과에 따라 `결제완료(PAID)`(성공) 또는 보상 후 `결제실패(PAYMENT_FAILED)`(실패) 로 단방향 전이한다. `PAID` 와 `PAYMENT_FAILED` 는 종료 상태이며 서로 전이하지 않는다 — 완료된 주문을 실패로, 실패한 주문을 완료로 전이하려는 시도는 거부된다.
- 주문 저장 트랜잭션 직후의 상태는 `결제대기` 이고, 상태 전이는 결제 결과 반영으로만 일어난다 — 결제 결과와 주문 상태는 어긋날 수 없다.
- 결제 실패 시 `PAYMENT_FAILED` 전이에는 보상이 함께 따른다 — 1단계에서 차감한 재고를 `Product.restoreStock(quantity)` 로 원복하고, 적용한 발급 쿠폰을 `UserCoupon.cancelUse()` 로 사용 취소(USED→AVAILABLE; 사용 전이면 멱등 no-op)한 뒤 주문을 `PAYMENT_FAILED` 로 마감한다. 상태 전이·보상은 결제 트랜잭션(`REQUIRES_NEW`) 진입 시 상태가 `결제대기` 일 때만 수행한다(멱등 가드 — 이중 청구·이중 보상 방지).
- 운영용 결제 게이트웨이는 항상 성공하는 mock 이라 정상 실행에서는 `PAYMENT_FAILED`·보상이 나타나지 않는다(실패 경로는 테스트로 커버).

**원자성(2단계)**
- 1단계 — 재고 차감 · 쿠폰 사용 · 주문 저장(`결제대기`) 은 하나의 트랜잭션 단위다. 전부 반영되거나 전부 취소되며, 어느 한 단계라도 실패하면 차감된 재고 · 소진된 쿠폰 · 저장 시도가 함께 롤백된다 — 부분 반영은 존재하지 않는다.
- 재고 차감과 쿠폰 사용의 상대 순서는 결과 정합성에 영향을 주지 않는다 — 둘 다 같은 트랜잭션 안에서 일어난다.
- 2단계 — 결제 호출은 1단계 트랜잭션이 **커밋된 뒤** 트랜잭션 밖에서 일어난다. 외부 호출을 DB 트랜잭션 안에 가두지 않는다(커넥션 장기 점유·"결제 성공인데 주문 롤백" 붕괴 방지).

**쿠폰 적용 (주문이 조율)**
- 주문에는 발급 쿠폰을 최대 1장만 적용한다.
- 적용 대상은 인증된 회원이 소유한 발급 쿠폰이어야 한다 — 타인 소유이거나 존재하지 않으면 적용할 수 없다.
- 발급 쿠폰은 최대 한 번 사용된다 — 이미 사용된 쿠폰은 주문에 적용할 수 없으며, 주문 성공 시 적용된 쿠폰은 즉시 사용 완료(소진) 되어 재사용이 불가하다.
- 만료된 쿠폰은 주문에 적용할 수 없다.
- 상품 합계가 쿠폰의 최소 주문 금액에 못 미치면 그 쿠폰을 적용할 수 없다.
- **쿠폰 적용의 조율은 본 도메인의 책임이다** — 발급 쿠폰을 동시성 안전하게 조회하고, 소유·사용 여부·만료·최소 주문 금액을 검증하며, 검증 통과 시 할인 계산과 사용 완료 전이를 호출하는 순서를 본 도메인이 소유한다.
- 단, **할인 계산식(정액/정률)·단일 사용·만료 판정 등 쿠폰 자체의 규칙은 Coupon 도메인 객체가 소유한다** — 본 도메인은 그 규칙을 직접 다시 구현하지 않고 Coupon 도메인 객체에 위임해 호출한다. (규칙의 세부는 [coupon/requirements.md](../coupon/requirements.md) 가 소유한다.)
- 재고 차감은 Product 도메인 객체에 위임한다 — 본 도메인은 차감 호출의 발기인이자 트랜잭션 경계의 단일 소유자다.

**회원 시각의 조회 정책**
- 회원 주문 목록의 기간 기준 시각은 주문 시각(생성 시각) 이다.
- 페이지 번호의 기본값은 `0`, 페이지 크기의 기본값은 `20` 이다. 미지정 입력은 기본값으로 해석되며, 허용 범위 밖 입력은 실패 분기로 분리된다.
- 회원은 자기 자신의 주문만 조회할 수 있다 — 타인의 주문 식별자가 회원 시각의 단일 조회로 들어오면 `FORBIDDEN`, 존재하지 않는 식별자는 `ORDER_NOT_FOUND` 로 분기된다 (두 의미가 구분된다).
- 회원의 주문 생성 · 목록 조회 · 단일 상세 조회는 모두 인증된 회원에게만 허용된다.

**어드민 시각의 조회 정책**
- 어드민 시각의 액터는 운영자다 — 회원은 본 시각의 데이터 주체로만 등장한다.
- 운영자에게는 회원 시각의 "자기 자신의 주문만 본다" 제약이 적용되지 않는다.
- 운영자 응답에는 운영 메타(회원 식별자, 회원 표시명, 결제 트랜잭션 식별자, 결제 결과 코드) 가 추가로 포함된다 — 회원 응답에는 포함되지 않는다.
- 회원 표시명은 User 도메인의 이름 마스킹 규칙(마지막 1글자를 `*` 로 치환) 을 그대로 적용한다.
- 카드 번호 · CVC 등 결제 민감 정보는 운영 메타에도 포함되지 않는다.
- 어드민 주문 목록의 정렬은 주문 시각 내림차순이며, 본 명세 범위에서 회원/기간/상태 필터를 지원하지 않는다.
- LDAP 식별 실패와 권한 부족은 응답에서 구분된다 — 식별 실패는 `UNAUTHORIZED` (401), 권한 부족은 `FORBIDDEN` (403).
- 어드민 시각은 GET only 다.
- 운영자의 조회 행위에 대한 감사 로그는 본 도메인이 책임지지 않는다 — 일반 액세스 로깅에 위임한다.

> 데이터 구조 결정은 [docs/design/04-erd.md](../../design/04-erd.md) 의 `orders`(상품 합계 · 할인 금액 · 결제 금액 · 적용 발급 쿠폰 식별자 · 결제 메타 컬럼 포함) · `order_lines` · `coupons` · `user_coupons` 를 참고한다.

## 6. 비기능 요구사항

- 동일 회원·동일 멱등 키의 동시 요청에서도 주문은 1건을 넘지 않는다 — 멱등 약속이 동시성에서 깨지지 않는다.
- 같은 발급 쿠폰으로 여러 기기에서 동시에 주문해도 그 쿠폰은 단 한 번만 사용된다 — 단일 사용 약속이 동시성에서 깨지지 않는다.
- 같은 상품에 동시 주문이 몰릴 때 재고 차감의 정합성이 보장된다 — 재고는 음수가 되지 않으며 Lost Update 가 발생하지 않는다.
- 결제 게이트웨이 호출은 DB 트랜잭션 밖에서 일어난다 — 외부 호출이 트랜잭션 커넥션을 점유하지 않는다.
- 모든 응답은 표준 응답 봉투(`ApiResponse<T>`) 로 감싼다.
- 주문 시각·기간 비교 등 시간 처리는 `Asia/Seoul` 기준으로 일관되게 처리한다.
