# 주문(Order) 도메인 요구사항 명세

> **상태**: **v0.2 초안**   
> **작성일**: 2026-05-20   
> **변경 이력**:
> - v0.2 (2026-05-20): 어드민 시각 흡수. 운영자(Admin) · LDAP 인증 · 운영 메타 · 회원 표시명 · 어드민 API 경로 어휘를 §1 에 추가하고, 어드민 주문 목록 조회·단일 상세 조회를 UC-4·UC-5 로 추가한다. 본인 제약 비적용 · 운영 메타 추가 노출 · 어드민 GET only · 감사 로깅 위임을 §5·§6 에 반영한다. 별도 `requirements-admin.md` 는 본 파일로 통합한다.
> - v0.1 (2026-05-20): 최초 초안. 주문 생성·결제 흐름(다수 라인 아이템, 가격 스냅샷, 재고 차감, 외부 결제 게이트웨이 협력, 보상 트랜잭션), 멱등 키 기반 중복 흡수, 기간·페이지 단위 목록 조회, 단일 상세의 권한 응답 정책을 정의한다. 쿠폰은 주문 입력의 선택 슬롯과 합계 반영까지만 박고, 발급·검증·소진의 책임 분리는 미해결로 둔다. 포인트 차감은 본 명세 범위 밖이다.

## 1. 유비쿼터스 언어

| 용어 | 의미 |
|---|---|
| **주문(Order)** | 회원의 구매 의사를 표현하는 거래 단위. 영문 표기는 `Order`. 정확히 한 명의 회원에 소속되고, 1개 이상의 주문 라인을 가진다. 주문 시각·합계 금액·상태를 가진다. |
| **주문 라인(OrderLine)** | 한 주문에 속한 한 상품-수량 단위. 상품 식별자·수량과 함께 **시점 가격(`unitPrice`)** · **시점 상품명(`productName`)** 의 스냅샷을 박아 보관한다. |
| **주문 상태(OrderStatus)** | 주문 생명주기의 단일 결과 상태. `결제완료(PAID)` 또는 `결제실패(PAYMENT_FAILED)` 두 값. 본 명세는 그 외 상태(취소·배송 등) 를 다루지 않는다. |
| **합계 금액(totalAmount)** | 주문 라인의 `unitPrice × 수량` 합산에 쿠폰 적용 결과가 반영된 최종 결제 대상 금액. 음수가 될 수 없다. |
| **회원(User)** | 회원 시각 유스케이스의 주체이자 모든 주문의 데이터 주체. User 도메인이 정의한 헤더 인증 절차를 그대로 사용한다. 비회원은 회원 시각의 어떤 흐름에도 진입할 수 없다. |
| **상품(Product)** | 주문 라인의 대상. Product 도메인이 정의하며, 본 도메인은 그 존재 여부·시점 가격·이름을 외부 의존으로 사용한다. |
| **재고(Stock)** | 상품의 잔여 수량. 외부 도메인의 개념이며, 주문 합의 시 본 도메인이 차감 호출의 발기인이 된다. 차감 실패는 주문 실패로 직결된다. |
| **외부 결제 게이트웨이(Payment Gateway)** | 결제 처리를 위임하는 외부 시스템. 본 명세에서는 추상 의존으로만 등장하며, 구체 PG·모킹 여부와 동기/비동기 방식은 본 명세에서 결정하지 않는다. |
| **쿠폰(Coupon)** | 주문 입력에 식별자로 동반될 수 있는 외부 도메인 개념. 본 명세는 입력 슬롯과 합계 금액 반영까지만 다루며, 발급·검증·소진의 책임 분리는 미해결로 둔다. |
| **헤더 인증** | User 도메인이 정의한 `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 기반 식별 절차. 회원 시각의 모든 유스케이스 진입 전 선행한다. |
| **멱등 키(Idempotency Key)** | 주문 생성 요청의 중복 흡수를 위해 클라이언트가 발급하는 식별 문자열. 같은 회원의 같은 키 재요청은 기존 주문과 동일한 응답으로 수렴한다. |
| **보상(Compensation)** | 외부 결제 실패 또는 쿠폰 적용 실패 시 앞서 차감한 재고(및 적용된 쿠폰 효과) 를 명시적으로 되돌리는 본 도메인의 책임. |
| **페이지(Page)** | 주문 목록의 분할 단위. `page` (0부터) 와 `size` 두 값으로 식별되는 offset 페이징을 사용한다. 미지정 시 `page=0` / `size=20` 이 적용된다. |
| **운영자(Admin)** | 회사 내부에서 LDAP 으로 식별된 사람. CS·정산·운영을 위해 본 도메인에 접근하는 액터이며, 어드민 시각 유스케이스의 주체다. 회원이 아니다. |
| **LDAP 인증** | 운영자의 식별·인가를 사내 LDAP 으로 처리하는 절차. 본 명세는 그 절차를 추상 의존으로만 둔다 — 구체 헤더·토큰·권한 키 정의는 본 명세에서 결정하지 않는다. |
| **운영 메타(Admin Metadata)** | 회원 응답에는 노출되지 않지만 운영자 응답에는 추가로 노출되는 정보 묶음 — 회원 식별자, 회원 표시명, 외부 결제 게이트웨이의 트랜잭션 식별자, 결제 결과 코드. |
| **회원 표시명** | 운영자 응답에 회원을 식별 가능하게 보여주기 위한 문자열. User 도메인의 이름 마스킹 규칙(마지막 1글자를 `*` 로 치환) 을 그대로 적용한다. |
| **어드민 API 경로** | 회원 API 와 분리된 게이트로, `/api-admin/v1/...` 로 노출된다. 어드민 시각 유스케이스는 모두 이 경로 아래에 있다. |

## 2. 유저 스토리

**회원 시각**
- 회원은 1개 이상의 상품-수량(라인 아이템) 을 담아 주문을 요청할 수 있다.
- 회원은 주문 시 쿠폰을 사용할 수 있다.
- 회원은 같은 멱등 키로 주문 요청을 여러 번 보내도 한 주문만 만들어짐을 기대할 수 있다.
- 회원은 자신의 주문 목록을 기간으로 조회할 수 있다 — 결제 실패 주문도 함께 보인다.
- 회원은 자신의 단일 주문 상세를 조회할 수 있다 — 주문 시점의 가격과 상품명이 그대로 보인다.
- 회원은 자기 자신의 주문만 조회할 수 있다 — 타인의 주문은 단일 조회로도 보지 못한다.

**운영자 시각**
- 운영자는 전사의 주문 목록을 페이지 단위로 조회할 수 있다 — 결제 실패 주문도 함께 본다.
- 운영자는 단일 주문 식별자로 그 주문의 상세를 조회할 수 있다 — 어느 회원의 주문이든 본인 한정 제약 없이 본다.
- 운영자는 주문 응답에서 회원 식별자와 운영 메타(결제 게이트웨이 트랜잭션 식별자·결과 코드 등) 를 함께 본다.
- 운영자는 카드 번호 · CVC 등 결제 민감 정보를 어떤 응답에서도 보지 않는다.

> 본 명세 범위 밖이다: 주문 취소, 포인트 차감/적립, 비회원 주문, 쿠폰의 발급·검증·소진 책임 분리, 외부 결제 게이트웨이의 구체 PG·모킹·동기/비동기 결정

## 3. 기능 요구사항

**회원 시각**
- 회원이 인증 헤더와 멱등 키, 1개 이상의 라인 아이템(상품 식별자 · 수량), 선택적 쿠폰 식별자를 담아 주문을 요청하면, 재고 차감과 외부 결제가 모두 성공한 경우에만 주문이 `결제완료` 로 확정된다.
- 회원이 같은 멱등 키로 주문을 다시 요청하면 신규 주문이 만들어지지 않고 기존 주문과 동일한 응답을 받는다.
- 회원이 라인 아이템에 존재하지 않는 상품 식별자를 담아 보내면 실패 응답을 받는다 — 어떤 라인도 처리되지 않고 주문이 만들어지지 않는다.
- 회원의 라인 아이템 중 하나라도 재고가 부족하면 실패 응답을 받는다 — 그 시점까지 차감된 재고는 모두 원복된다.
- 외부 결제 게이트웨이의 결제가 실패하면 회원은 실패 응답을 받는다 — 차감된 재고는 모두 원복된다.
- 회원이 자신의 주문 목록을 `startAt` · `endAt` · 페이지 번호 · 페이지 크기와 함께 요청하면, 기간 안의 자신의 주문이 주문 시각 내림차순으로 페이지 단위로 응답된다.
- 회원 주문 목록 응답에는 `결제완료` 와 `결제실패` 주문이 모두 포함된다.
- 회원이 페이지 번호 또는 페이지 크기를 허용 범위 밖으로 보내면 실패 응답을 받는다.
- 회원이 페이지 번호 · 페이지 크기를 지정하지 않으면 기본값(`page=0` / `size=20`) 으로 응답된다.
- 회원이 단일 주문 식별자로 상세를 요청하면 주문 정보(주문 시각 · 상태 · 합계) 와 라인 아이템(시점 가격 · 시점 상품명 · 수량) 을 응답받는다.
- 회원이 타인의 주문 식별자로 단일 조회를 요청하면 권한 실패 응답을 받는다 — 존재하지 않는 식별자와 응답이 구분된다.
- 회원이 존재하지 않는 주문 식별자로 단일 조회를 요청하면 "찾을 수 없음" 실패 응답을 받는다.
- 인증되지 않은 회원 요청은 회원 시각의 어느 흐름에도 진입할 수 없다.

**운영자 시각**
- 운영자가 LDAP 인증과 페이지 번호 · 페이지 크기를 담아 주문 목록을 요청하면, 전사 주문이 주문 시각 내림차순으로 페이지 단위로 응답된다.
- 어드민 주문 목록 응답에는 `결제완료` 와 `결제실패` 주문이 모두 포함된다.
- 운영자가 페이지 번호 · 페이지 크기를 지정하지 않으면 기본값(`page=0` / `size=20`) 으로 응답된다.
- 운영자가 페이지 번호 또는 페이지 크기를 허용 범위 밖으로 보내면 실패 응답을 받는다.
- 운영자가 단일 주문 식별자로 상세를 요청하면 주문 정보와 라인 아이템과 운영 메타를 함께 응답받는다.
- 운영자가 존재하지 않는 주문 식별자로 단일 조회를 요청하면 "찾을 수 없음" 실패 응답을 받는다.
- 운영자는 본인 한정 제약 없이 어느 회원의 주문이든 조회할 수 있다 — 회원 시각의 권한 실패 분기는 어드민 시각에서 발생하지 않는다.
- LDAP 인증에 실패한 요청은 어드민 시각의 어느 흐름에도 진입할 수 없다.
- LDAP 으로 식별은 됐으나 주문 조회 권한이 없는 운영자의 요청은 권한 실패 응답을 받는다.
- 회원의 카드 번호 · CVC 등 결제 민감 정보는 운영 메타에도 포함되지 않는다.

## 4. 유스케이스 흐름

### 공통 사전 단계 A — 헤더 인증 (회원 시각)

UC-1, UC-2, UC-3 의 Main Flow 진입 전에 항상 선행한다.

1. 시스템은 `X-Loopers-LoginId` 와 `X-Loopers-LoginPw` 두 헤더의 존재를 확인한다.
2. 시스템은 LoginId 로 회원을 조회하고, 저장된 비밀번호와 헤더 값의 일치를 확인한다.
3. 두 검사가 모두 통과하면 해당 회원을 컨텍스트로 두고 UC 본 흐름으로 진입한다.

**Exception**
- **E0** (사전 단계 어디서든): 헤더 누락 / 회원 없음 / 비밀번호 불일치 중 하나라도 발생 → `UNAUTHORIZED` (401), 상태 변경 없음. 어느 원인인지 응답에서 구분하지 않는다.

### 공통 사전 단계 B — LDAP 인증 (어드민 시각)

UC-4, UC-5 의 Main Flow 진입 전에 항상 선행한다. LDAP 식별·인가의 구체 절차는 본 명세 범위 밖이며, 본 명세는 그 절차가 다음 두 결과 중 하나를 만들어낸다고만 본다 — 성공 시 운영자 컨텍스트가 확정된 상태로 본 흐름에 진입하고, 실패 시 본 흐름에 진입하지 않는다.

**Exception**
- **E0-a** (사전 단계): LDAP 식별 실패(식별자 누락 / 자격 증명 실패 등) → `UNAUTHORIZED` (401), 상태 변경 없음. 어느 원인인지 응답에서 구분하지 않는다.
- **E0-b** (사전 단계): LDAP 식별은 통과했으나 주문 조회 권한 부족 → `FORBIDDEN` (403), 상태 변경 없음.

### UC-1. 주문 생성 (회원)

> 공통 사전 단계 A (헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 멱등 키, 1개 이상의 라인 아이템(상품 식별자 · 수량), 선택적 쿠폰 식별자를 담아 주문 생성을 요청한다.
2. 시스템은 인증된 회원과 멱등 키의 조합으로 기존 주문이 있는지 확인한다.
3. 시스템은 각 라인 아이템의 상품 존재를 확인하고, 시점 가격과 시점 상품명을 스냅샷한다.
4. 시스템은 각 라인 아이템의 재고를 차감한다.
5. 시스템은 쿠폰 식별자가 있으면 쿠폰의 적용 결과를 합계 금액에 반영한다.
6. 시스템은 외부 결제 게이트웨이에 합계 금액의 결제를 요청한다.
7. 시스템은 주문을 `결제완료(PAID)` 로 확정하고, `200 OK` 와 함께 주문 식별자 · 주문 시각 · 상태 · 합계 금액 · 라인 아이템 스냅샷을 응답한다.

**Alternate Flow — 같은 멱등 키 재요청** (2 → 7a)
- 7a. 시스템은 새 주문을 만들지 않고, 기존 주문과 동일한 응답을 그대로 돌려준다 (멱등 수렴).

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (1): 멱등 키가 비어 있거나, 라인 아이템이 비어 있거나, 어떤 라인의 수량이 1 미만 → `BAD_REQUEST` (400), 상태 변경 없음.
- **E2** (3): 라인 아이템 중 하나라도 상품이 존재하지 않음 → `PRODUCT_NOT_FOUND` (404), 어떤 라인도 처리되지 않고 주문 생성 없음.
- **E3** (4): 라인 아이템 중 하나라도 재고가 부족 → `OUT_OF_STOCK` (409), 그 시점까지 차감된 재고는 모두 원복(보상), 주문 생성 없음.
- **E4** (5): 쿠폰 적용 실패(존재하지 않음 / 사용 불가) → `COUPON_NOT_APPLICABLE` (400), 차감된 재고는 모두 원복(보상), 주문 생성 없음.
- **E5** (6): 외부 결제 게이트웨이 결제 실패 → 차감된 재고와 적용된 쿠폰 효과는 모두 원복(보상), 주문은 `결제실패(PAYMENT_FAILED)` 상태로 처리된다 (`결제실패` 주문 행의 잔존 여부는 **TBD**). 실패를 에러 응답으로 표면화할 경우의 HTTP 상태는 고정값(402)을 쓰지 않고 실패 원인에 따라 결정한다 — 결제 수단 오류는 `BAD_REQUEST` (400), 이미 결제 완료된 주문은 `CONFLICT` (409) 등.

### UC-2. 내 주문 목록 조회 (회원)

> 공통 사전 단계 A (헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 `startAt` (선택) · `endAt` (선택) · 페이지 번호(선택) · 페이지 크기(선택) 를 담아 주문 목록을 요청한다.
2. 시스템은 페이지 번호 · 페이지 크기가 허용 범위 안에 있는지 검증한다.
3. 시스템은 미지정 입력을 기본값으로 대체한다 — 페이지 번호는 `0`, 페이지 크기는 `20`.
4. 시스템은 인증된 회원의 주문 중 주문 시각이 기간 안에 든 것들을 주문 시각 내림차순으로 페이지 단위로 응답한다.
5. 응답 각 항목은 주문 식별자 · 주문 시각 · 상태 · 합계 금액 · 라인 아이템 요약을 포함하며, `결제실패` 주문도 함께 포함된다.

**Alternate Flow — 결과가 비어 있음** (4 → 4a)
- 4a. 시스템은 빈 목록과 페이지 메타로 응답한다 (실패가 아니다).

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (2): 페이지 번호가 음수이거나 페이지 크기가 허용 범위를 벗어남 → `BAD_REQUEST` (400), 상태 변경 없음.

### UC-3. 단일 주문 상세 조회 (회원)

> 공통 사전 단계 A (헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 주문 식별자로 단일 주문 상세를 요청한다.
2. 시스템은 해당 주문의 존재를 확인한다.
3. 시스템은 그 주문의 소유 회원이 인증된 회원과 일치하는지 확인한다.
4. 시스템은 주문 정보(주문 식별자 · 주문 시각 · 상태 · 합계 금액) 와 라인 아이템(상품 식별자 · 시점 상품명 · 시점 가격 · 수량) 을 응답한다.

**Alternate Flow** — 없음.

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (2): 주문 식별자가 존재하지 않음 → `ORDER_NOT_FOUND` (404), 상태 변경 없음.
- **E2** (3): 주문의 소유 회원이 인증된 회원과 다름 → `FORBIDDEN` (403), 상태 변경 없음. (E1 과 응답이 분리되어 "존재 여부" 와 "권한 없음" 의 의미가 구분된다.)

### UC-4. 어드민 주문 목록 조회 (운영자)

> 공통 사전 단계 B (LDAP 인증) 를 먼저 통과한다.

**Main Flow**
1. 운영자가 페이지 번호(선택) · 페이지 크기(선택) 를 담아 주문 목록을 요청한다.
2. 시스템은 페이지 번호 · 페이지 크기가 허용 범위 안에 있는지 검증한다.
3. 시스템은 미지정 입력을 기본값(`page=0` / `size=20`) 으로 대체한다.
4. 시스템은 전체 주문을 주문 시각 내림차순으로 페이지 단위로 응답한다.
5. 응답 각 항목은 주문 식별자 · 주문 시각 · 상태 · 합계 금액 · 라인 아이템 요약 · 운영 메타(회원 식별자, 회원 표시명, 외부 결제 트랜잭션 식별자, 결제 결과 코드) 를 포함하며, `결제실패` 주문도 함께 포함된다.

**Alternate Flow — 결과가 비어 있음** (4 → 4a)
- 4a. 시스템은 빈 목록과 페이지 메타로 응답한다 (실패가 아니다).

**Alternate Flow — 결제 실패 주문이 외부 트랜잭션 식별자를 가지지 않음** (5 → 5a)
- 5a. 시스템은 해당 항목의 외부 결제 트랜잭션 식별자·결제 결과 코드를 빈 값으로 응답한다 (실패가 아니다).

**Exception Flow**
- E0-a / E0-b 에 위임 (LDAP 인증·인가 실패).
- **E1** (2): 페이지 번호가 음수이거나 페이지 크기가 허용 범위를 벗어남 → `BAD_REQUEST` (400), 상태 변경 없음.

### UC-5. 어드민 단일 주문 상세 조회 (운영자)

> 공통 사전 단계 B (LDAP 인증) 를 먼저 통과한다.

**Main Flow**
1. 운영자가 주문 식별자로 단일 주문 상세를 요청한다.
2. 시스템은 해당 주문의 존재를 확인한다.
3. 시스템은 주문 정보(주문 식별자 · 주문 시각 · 상태 · 합계 금액) 와 라인 아이템(상품 식별자 · 시점 상품명 · 시점 가격 · 수량) 과 운영 메타를 응답한다.

**Alternate Flow** — 없음. 운영자는 어느 회원의 주문이든 볼 수 있으므로 UC-3 의 "타인의 주문" 분기가 본 흐름에는 존재하지 않는다.

**Exception Flow**
- E0-a / E0-b 에 위임 (LDAP 인증·인가 실패).
- **E1** (2): 주문 식별자가 존재하지 않음 → `ORDER_NOT_FOUND` (404), 상태 변경 없음.

## 5. 도메인 규칙 / 불변식

**모델·상태·합의**
- 주문은 정확히 한 명의 회원에 속한다 — 회원 없는 주문은 존재하지 않는다.
- 주문은 1개 이상의 라인 아이템을 가진다 — 빈 주문은 존재하지 않는다.
- 라인 아이템의 수량은 1 이상의 정수다.
- 라인 아이템은 시점 가격(`unitPrice`) 과 시점 상품명(`productName`) 의 스냅샷을 보관한다 — 이후 카탈로그 변경이 과거 주문의 합계나 표시명을 바꾸지 않는다.
- 주문의 합계 금액은 라인 아이템의 `unitPrice × 수량` 합산에 쿠폰 적용 결과가 반영된 최종 결제 대상 금액이다. 음수가 될 수 없다.
- 주문 상태는 `결제완료(PAID)` 또는 `결제실패(PAYMENT_FAILED)` 두 값으로 한정된다. 본 도메인은 그 외 전이(취소·배송 등) 를 다루지 않는다.
- 주문 생성은 멱등 키로 수렴한다 — 같은 회원의 같은 멱등 키 재요청은 신규 주문을 만들지 않고 기존 주문과 동일한 응답을 돌려준다.
- 주문 생성 흐름은 다음 순서로 합의한다: 상품 존재 확인 → 재고 차감 → (쿠폰 적용) → 외부 결제. 후속 단계의 실패는 앞서 차감한 재고와 적용된 쿠폰 효과를 명시적으로 되돌리는 보상 책임을 본 도메인이 진다.
- 외부 결제 게이트웨이의 결제 결과와 주문 상태는 어긋날 수 없다 — 결제 성공은 `결제완료`, 결제 실패는 `결제실패` 로만 귀결된다.
- 재고 차감 · 외부 결제 호출 · 쿠폰 적용은 모두 외부 도메인의 책임이다 — 본 도메인은 호출의 발기인이자 결과에 따른 자기 상태(주문 상태) 의 일관성과 보상의 책임을 진다.
- 쿠폰은 주문 입력의 선택 슬롯과 합계 금액의 반영으로만 본 도메인에 등장한다 — 발급·검증·소진의 책임 분리는 본 명세에서 결정하지 않는다.

**회원 시각의 조회 정책**
- 회원 주문 목록의 기간 기준 시각은 주문 시각(생성 시각) 이다 — `결제실패` 주문도 목록에 포함된다.
- 페이지 번호의 기본값은 `0`, 페이지 크기의 기본값은 `20` 이다. 미지정 입력은 기본값으로 해석되며, 허용 범위 밖 입력은 실패 분기로 분리된다.
- 회원은 자기 자신의 주문만 조회할 수 있다 — 타인의 주문 식별자가 회원 시각의 단일 조회로 들어오면 `FORBIDDEN` 으로 분기되고, 존재하지 않는 식별자는 `ORDER_NOT_FOUND` 로 분기된다 (두 의미가 구분된다).
- 회원의 주문 생성 · 목록 조회 · 단일 상세 조회는 모두 인증된 회원에게만 허용된다 — 비회원/미인증 요청은 회원 시각 흐름에 진입할 수 없다.

**어드민 시각의 조회 정책**
- 어드민 시각의 액터는 운영자다 — 회원은 본 시각의 데이터 주체로만 등장한다.
- 운영자에게는 회원 시각의 "자기 자신의 주문만 본다" 제약이 적용되지 않는다 — 회원 시각의 `FORBIDDEN` 분기는 어드민 시각에서 발생하지 않는다.
- 운영자 응답에는 운영 메타(회원 식별자, 회원 표시명, 외부 결제 트랜잭션 식별자, 결제 결과 코드) 가 추가로 포함된다 — 회원 응답에는 포함되지 않는다.
- 회원 표시명은 User 도메인의 이름 마스킹 규칙(마지막 1글자를 `*` 로 치환) 을 그대로 적용한다 — 운영 화면에서도 회원의 본명 마지막 글자는 노출되지 않는다.
- 카드 번호 · CVC 등 결제 민감 정보는 운영 메타에도 포함되지 않는다 — 어드민 응답 역시 그 규칙의 예외가 아니다.
- 어드민 주문 목록의 정렬은 주문 시각 내림차순이며, `결제완료` 와 `결제실패` 주문이 함께 포함된다.
- 어드민 주문 목록 조회는 본 명세 범위에서 회원/기간/상태 필터를 지원하지 않는다 — 단순 페이지·크기 입력만 받는다.
- LDAP 식별 실패와 권한 부족은 응답에서 구분된다 — 식별 실패는 `UNAUTHORIZED` (401), 권한 부족은 `FORBIDDEN` (403). LDAP 인증의 구체 절차는 본 도메인 책임이 아니다.
- 어드민 시각은 GET only 다 — 운영자의 강제 주문 취소·환불·결제 재시도는 본 명세 범위 밖이다.
- 운영자의 조회 행위에 대한 감사 로그는 본 도메인이 책임지지 않는다 — 일반 액세스 로깅에 위임한다.

## 6. 비기능 요구사항

- 동일 회원·동일 멱등 키의 동시 요청에서도 주문은 1건을 넘지 않는다 — 멱등 약속이 동시성에서 깨지지 않는다.
- 같은 상품에 동시 주문이 몰릴 때 재고 차감의 정합성은 외부 재고 도메인이 보장한다 — 본 도메인은 차감 실패의 보상 책임만 진다.
