# 결제(Payment) 도메인 요구사항 명세

> **상태**: v0.1 초안
> **작성일**: 2026-06-23
> **변경 이력**:
> - v0.1 (2026-06-23): 최초 초안. Round 6 — 외부 PG(`pg-simulator`) **비동기 결제 연동**과 **회복 전략(Timeout · CircuitBreaker · Fallback · 폴링 복구)** 을 정의한다. 결제 진입점을 명시 API(`POST /api/v1/payments`)로 일원화하고, 콜백을 주 경로 · 폴링을 안전망으로 둔다. 영구 실패(잘못된 카드 · 한도 초과)는 즉시 실패 확정, 일시 장애(통신 실패 · 타임아웃 · CB Open)는 처리 중 유지 후 복구로 가른다. 결제 전 생애주기(요청 · 승인 · 실패 · 취소/환불)를 본 도메인이 소유하며, 주문(Order) 은 결제대기 주문 생성까지만 책임진다([order/requirements.md](../order/requirements.md) v0.6 동기화).

## 1. 유비쿼터스 언어

| 용어 | 의미 |
|---|---|
| **결제(Payment)** | 한 주문에 대한 결제 시도와 그 결과의 source of truth. 영문 표기는 `Payment`. 정확히 하나의 주문에 귀속되며, **1주문 1결제** 다. 결제 금액 · 상태 · 외부 거래 식별자 · 실패 사유 · 시각(요청 · 승인 · 취소) 을 가진다. |
| **결제 상태(PaymentStatus)** | 결제 생명주기의 상태. `요청(REQUESTED)` 은 외부 PG 에 결제를 요청하고 결과를 기다리는 처리 중 상태이며, 결과에 따라 `승인(APPROVED)` 또는 `실패(FAILED)` 로, 사후 환불 시 `취소(CANCELED)` 로 전이한다. |
| **외부 결제 시스템(PG)** | 결제 처리를 위임하는 외부 의존(`pg-simulator`). **비동기** 다 — 요청 접수와 실제 처리(승인/거절)가 분리되어 있다. 요청은 100~500ms 안에 접수되고, 처리는 1~5초 뒤에 확정되며, 그 결과가 콜백 또는 상태 확인 API 로 도착한다. |
| **결제 요청(Charge)** | 외부 PG 에 결제를 위임하는 행위. 즉시 성공/실패를 확정하지 않고, **요청 접수(거래 식별자 + 처리 중)** 만 돌려받는다. |
| **거래 식별자(transactionKey)** | 외부 PG 가 요청 접수 시 발급하는 식별자. 콜백·폴링에서 결제건을 매칭하는 기준이다. |
| **카드 정보(Card)** | 회원이 결제 시 제출하는 카드 종류와 카드 번호. 카드 번호는 민감정보로 어떤 응답·로그에도 노출되지 않는다. |
| **콜백(Callback)** | 외부 PG 가 처리 결과를 우리 시스템의 통지 주소로 알려 오는 채널. 결과 반영의 **주 경로** 다. |
| **상태 폴링(Polling)** | 처리 중 결제를 외부 PG 의 상태 확인 API 로 조회해 최종 상태로 수렴시키는 **안전망**. 주기적(스케줄) · 수동(운영자) 으로 트리거된다. |
| **정산(Settle)** | 외부 PG 의 처리 결과(성공/실패)를 결제·주문 상태에 반영하는 행위. **멱등** 하다 — 콜백·폴링이 겹쳐 도착해도 한 번만 반영된다. |
| **환불(Refund)** | 승인된 결제를 외부 PG 에 취소 요청하는 행위. 거래 식별자로 멱등 처리한다(재시도 시 이중 환불 방지). |
| **Fallback** | 외부 PG 의 지연·장애로 결과를 즉시 알 수 없을 때, 내부 시스템이 멈추지 않고 정상 응답하도록 두는 대응. |
| **영구 실패 / 일시 장애** | **영구 실패** 는 카드 자체의 문제(잘못된 카드 · 한도 초과) — 재시도해도 동일하므로 즉시 실패로 확정한다. **일시 장애** 는 통신 실패 · 타임아웃 · 회로 차단(CB Open) — 외부는 살아 있을 수 있으므로 처리 중으로 두고 복구한다. |
| **타임아웃(Timeout)** | 외부 PG 호출이 정한 시간 안에 응답하지 않으면 실패로 간주하고 호출 스레드를 풀어 주는 한계 시간. 연결(connect) 과 응답(read) 을 분리해 둔다. |
| **회로 차단기(Circuit Breaker)** | 외부 PG 가 반복 실패하면 호출을 일시 차단(Open)해 장애 확산을 막는 장치. `Closed`(정상) · `Open`(차단) · `Half-Open`(일부 시도) 상태를 가진다. |
| **보상(Compensation)** | 결제가 실패로 확정될 때, 주문 생성 단계에서 차감했던 재고와 사용한 쿠폰을 원복하는 행위. 주문(Order) 도메인의 자원을 되돌리는 협력이다. |
| **주문(Order)** | 결제의 귀속 대상. Order 도메인이 정의하며, 본 도메인은 그 결제대기 주문에 대해 결제를 수행하고 결과에 따라 주문 상태(결제완료/결제실패) 전이와 보상을 조율한다. |
| **헤더 인증** | User 도메인이 정의한 `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 기반 식별 절차. 회원의 결제 요청 진입 전 선행한다. |
| **운영자(Admin)** | 처리 중에 멈춘 결제의 상태 복구를 수동으로 트리거할 수 있는 액터. |

## 2. 유저 스토리

**회원 시각**
- 회원은 자신의 결제대기 주문에 카드 종류·번호를 입력해 결제를 요청할 수 있다.
- 회원은 결제를 요청하면 결과를 끝까지 기다리지 않고 즉시 "결제 접수(처리 중)" 응답을 받을 수 있다.
- 회원은 잘못된 카드 또는 한도 초과 카드로 결제하면 결제가 실패함을 안내받을 수 있다.
- 회원은 외부 결제 시스템이 느리거나 장애여도 우리 시스템이 멈추지 않고 정상 응답함을 기대할 수 있다.
- 회원은 같은 주문에 결제를 여러 번 요청해도 이중으로 청구되지 않음을 기대할 수 있다.
- 회원은 결제 결과(승인/실패)를 이후 주문 조회에서 확인할 수 있다.

**운영자/시스템 시각**
- 운영자는 콜백이 오지 않아 처리 중에 멈춘 결제의 상태를 수동으로 복구할 수 있다.
- 운영자는 외부에서는 결제됐지만 우리 시스템에는 반영되지 않은 결제건이 복구되어 최종 상태로 수렴함을 기대할 수 있다.

> 본 명세 범위 밖이다: 결제 수단의 등록·보관(매 요청 시 카드 입력), 부분 결제·복합 결제(1주문 1결제), 주문 취소 흐름 자체의 진입점(Order 도메인 소유 — 본 도메인은 환불·취소 능력만 제공).

## 3. 기능 요구사항

사용자 관점에서 관찰 가능한 동작만 적는다.

- 회원이 인증 헤더와 결제대기 주문 식별자, 카드 종류·번호를 담아 결제를 요청하면, 결과를 기다리지 않고 즉시 "결제 접수(처리 중)" 응답을 받는다.
- 결제 결과는 비동기로 확정되며, 회원은 이후 주문 조회에서 결제완료 또는 결제실패를 확인한다.
- 회원이 잘못된 카드 또는 한도 초과 카드로 결제하면 결제가 실패로 확정되고, 그 주문은 결제실패가 되며 차감됐던 재고와 사용한 쿠폰이 원복된다.
- 외부 결제 시스템이 지연·장애여도 회원의 결제 요청은 즉시 응답을 받는다 — 결제는 처리 중으로 남아 이후 복구된다.
- 회원이 자신 소유가 아닌 주문 식별자로 결제를 요청하면 권한 실패 응답을 받는다 — 존재하지 않는 주문과 응답이 구분된다.
- 회원이 결제대기가 아닌 주문(이미 결제완료·결제실패·취소된 주문)에 결제를 요청하면 실패 응답을 받는다.
- 회원이 같은 주문에 진행 중(처리 중·승인) 결제가 있는 동안 다시 결제를 요청하면 신규 결제가 만들어지지 않고 기존 접수 상태로 응답된다.
- 콜백이 도착하지 않은 결제는 일정 주기 또는 운영자의 수동 트리거로 외부 상태를 확인해 최종 상태로 수렴한다.
- 요청이 타임아웃으로 실패한 것처럼 보여도, 외부에 실제 접수·성공된 결제는 조회를 통해 정상 반영된다.
- 회원의 카드 번호는 어떤 응답·로그에도 노출되지 않는다.

## 4. 유스케이스 흐름

### 공통 사전 단계 — 헤더 인증 (회원 시각)

UC-1 의 Main Flow 진입 전에 항상 선행한다.

1. 시스템은 `X-Loopers-LoginId` 와 `X-Loopers-LoginPw` 헤더의 존재를 확인한다.
2. 시스템은 LoginId 로 회원을 조회하고, 저장된 비밀번호와 헤더 값의 일치를 확인한다.
3. 두 검사가 모두 통과하면 해당 회원을 컨텍스트로 두고 본 흐름으로 진입한다.

**Exception**
- **E0** (사전 단계 어디서든): 헤더 누락 / 회원 없음 / 비밀번호 불일치 → `UNAUTHORIZED` (401), 상태 변경 없음.

### UC-1. 결제 요청 (회원)

> 공통 사전 단계(헤더 인증) 를 먼저 통과한다.
> 외부 PG 호출은 트랜잭션·락 밖에서 일어난다. 호출 *전에* 결제를 `요청(REQUESTED)` 으로 먼저 영속(커밋)해, 호출 중 중단되어도 결제건이 추적·복구 가능하게 둔다.

**Main Flow**
1. 회원이 결제대기 주문 식별자, 카드 종류, 카드 번호를 담아 결제를 요청한다.
2. 시스템은 그 주문이 존재하고 인증된 회원의 소유이며 결제대기 상태인지 확인한다.
3. 시스템은 그 주문에 진행 중(처리 중·승인) 결제가 없음을 확인하고, 주문의 결제 금액으로 결제를 `요청(REQUESTED)` 으로 영속한다.
4. 시스템은 트랜잭션·락 밖에서 외부 PG 에 결제를 요청한다 — 카드 정보·결제 금액·결과 통지 주소를 전달한다.
5. 시스템은 외부 PG 가 접수한 거래 식별자를 결제에 기록하고, 즉시 "결제 접수(처리 중)" 를 응답한다. 최종 결과는 비동기로 확정된다.

**Alternate Flow — 진행 중 결제 재요청** (3 → 3a)
- 3a. 같은 주문에 이미 진행 중(처리 중·승인) 결제가 있으면, 시스템은 신규 결제를 만들지 않고 기존 접수 상태로 응답한다(이중 청구 방지).

**Alternate Flow — 일시 장애 Fallback** (4 → 4a)
- 4a. 외부 호출이 타임아웃되거나 통신 실패하거나 회로가 차단(CB Open) 상태이면, 시스템은 결제를 `요청(REQUESTED)` 으로 둔 채 스레드를 점유하지 않고 즉시 "결제 접수(처리 중)" 를 응답한다. 거래 식별자를 받지 못한 경우, 이후 폴링이 주문 식별자로 외부 상태를 조회해 수렴시킨다.

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (1): 카드 번호 형식 위반 / 카드 종류 누락 / 주문 식별자 누락 → `BAD_REQUEST` (400), 상태 변경 없음.
- **E2** (2): 주문 식별자가 존재하지 않음 → `ORDER_NOT_FOUND` (404), 상태 변경 없음.
- **E3** (2): 주문의 소유 회원이 인증된 회원과 다름 → `FORBIDDEN` (403), 상태 변경 없음. (E2 와 응답이 분리되어 "존재 여부" 와 "권한 없음" 의 의미가 구분된다.)
- **E4** (2): 주문이 결제대기 상태가 아님(이미 결제완료·결제실패·취소) → `CONFLICT` (409), 상태 변경 없음.

### UC-2. 결제 결과 콜백 수신·정산 (외부 PG → 시스템)

> 외부 PG 가 처리(1~5초) 를 마친 뒤 결과 통지 주소로 콜백한다. 결과 반영의 주 경로다.

**Main Flow**
1. 외부 PG 가 거래 식별자·처리 상태·사유를 담아 결과 통지 주소로 콜백한다.
2. 시스템은 거래 식별자로 대상 결제를 동시성 안전하게 조회한다.
3. 시스템은 그 결제가 아직 처리 중(요청) 일 때만 결과를 반영한다.
   - **성공** → 결제를 `승인(APPROVED)` 으로 전이하고, 그 주문을 결제완료로 전이시킨다.
   - **실패** → 결제를 `실패(FAILED)` 로 전이하고, 보상(차감 재고·사용 쿠폰 원복) 후 그 주문을 결제실패로 전이시킨다.
4. 시스템은 콜백 수신을 정상 응답한다.

**Alternate Flow — 이미 정산된 결제** (3 → 3a)
- 3a. 결제가 이미 승인·실패·취소 상태면, 시스템은 콜백을 멱등하게 무시하고 정상 응답한다(같은 콜백 2회, 콜백과 폴링 경합에도 정산은 한 번만).

**Exception Flow**
- **E1** (2): 거래 식별자에 해당하는 결제가 우리 시스템에 없음 → 정산 없이 수신만 기록하고 정상 응답, 상태 변경 없음. (외부 통지를 거부하지 않고 흡수한다.)

### UC-3. 결제 상태 폴링 복구 (시스템 스케줄러 / 운영자)

> 콜백 미수신·요청 타임아웃으로 처리 중에 멈춘 결제를 외부 상태 확인으로 수렴시키는 안전망. 콜백이 정상이면 이 경로는 아무 일도 하지 않는다.

**Main Flow**
1. 시스템 스케줄러(주기) 또는 운영자(수동) 가 처리 중(요청) 결제들의 상태 복구를 트리거한다.
2. 시스템은 각 처리 중 결제에 대해 거래 식별자로 외부 PG 의 상태를 조회한다.
3. 외부 상태가 확정(성공/실패) 이면 시스템은 UC-2 와 동일하게 정산한다(멱등).
4. 외부 상태가 아직 처리 중이면 시스템은 그 결제를 다음 주기로 미룬다.

**Alternate Flow — 거래 식별자 미확보** (2 → 2a)
- 2a. 요청 타임아웃으로 거래 식별자를 받지 못한 결제는, 주문 식별자로 외부 결제건을 조회해 매칭한 뒤 동일하게 정산한다("타임아웃됐지만 외부는 성공" 케이스가 결제완료로 수렴한다).

**Exception Flow**
- **E1** (2): 외부 상태 확인 호출도 실패·타임아웃 → 이번 주기는 건너뛰고 다음 주기에 재시도, 상태 변경 없음.
- **E2** (2): 외부에 해당 결제건이 끝내 존재하지 않음(요청이 접수조차 안 됨) → **TBD**: 처리 중 결제의 만료/실패 확정 정책(아래 미해결).

### UC-4. 결제 취소·환불

> 주문 취소 흐름(진입점은 Order 도메인 소유, 본 명세 범위 밖) 이 결제 취소를 요청할 때 본 도메인이 수행하는 흐름. 외부 환불 호출도 트랜잭션·락 밖에서 일어난다.

**Main Flow**
1. 주문 취소 흐름이 대상 결제의 취소를 요청한다.
2. 시스템은 결제가 승인(APPROVED) 이면 트랜잭션·락 밖에서 외부 PG 에 환불을 요청한다(거래 식별자로 멱등).
3. 시스템은 결제를 `취소(CANCELED)` 로 전이한다.

**Alternate Flow — 청구 전 취소** (2 → 2a)
- 2a. 결제가 처리 중(요청) 상태면 아직 청구 전이므로, 외부 환불 없이 곧바로 `취소(CANCELED)` 로 전이한다.

**Exception Flow**
- **E1** (3): 결제가 이미 취소됨 → 멱등 no-op, 정상 응답, 추가 상태 변경 없음.
- **E2** (3): 실패(FAILED) 결제에 대한 취소 시도 → `CONFLICT` (409), 상태 변경 없음. (실패한 결제는 취소로 전이할 수 없다.)

## 5. 도메인 규칙 / 불변식

**모델·귀속**
- 결제는 정확히 하나의 주문에 귀속된다 — 주문 없는 결제는 존재하지 않는다.
- 한 주문에 진행 중(처리 중·승인) 결제는 최대 1건이다 — **1주문 1결제**.
- 결제 금액은 그 주문의 결제 금액과 일치한다.
- 결제는 외부 거래 식별자를 보관한다 — 요청 접수 시점에 확보하며, 콜백·폴링의 매칭 기준이다.

**상태 전이**
- 결제 상태는 `요청(REQUESTED)` 에서 출발해 `승인(APPROVED)`(성공) 또는 `실패(FAILED)`(실패) 로, 또는 사후 `취소(CANCELED)` 로 전이한다.
- 승인된 결제만 환불 대상이며, 환불 시 `취소` 로 전이한다. 처리 중 결제의 취소는 청구 전이므로 환불 없이 `취소` 로 전이한다.
- `실패` 는 종료 상태다 — 승인·취소로 뒤집을 수 없다. `취소` 도 종료 상태다.
- 처리 중이 아닌 결제에 결과를 다시 반영하려는 시도는 거부되거나 멱등하게 무시된다.

**비동기 정산·멱등**
- 정산은 결제가 처리 중(요청) 일 때만 한 번 반영된다 — 같은 콜백 2회, 콜백과 폴링이 겹쳐 도착해도 결과는 한 번만 확정된다.
- 콜백을 주 경로로, 폴링을 안전망으로 둔다 — 콜백이 도착하지 않아도 처리 중 결제는 폴링으로 유한 시간 안에 최종 상태로 수렴한다.
- 요청 타임아웃으로 거래 식별자를 확보하지 못해도, 주문 식별자로 외부 상태를 조회해 정합성을 수렴시킨다 — "타임아웃됐지만 외부는 성공" 은 결제완료로 수렴해야 한다.

**Fallback — 영구 실패 vs 일시 장애**
- 영구 실패(잘못된 카드 · 한도 초과) 는 즉시 `실패(FAILED)` 로 확정한다 — 재시도해도 결과가 같기 때문이다.
- 일시 장애(통신 실패 · 타임아웃 · 회로 차단) 는 결제를 `요청(REQUESTED)`(처리 중) 으로 유지한다 — 외부 일시 장애로 결제가 거짓 실패가 되어서는 안 된다. 이후 복구 경로가 최종 상태를 확정한다.

**외부 호출 격리**
- 외부 PG 호출(결제 요청 · 환불) 은 어떤 DB 트랜잭션·락 안에도 두지 않는다 — 외부 지연이 커넥션을 장기 점유해 전체로 전파되지 않게 한다.
- 외부 호출은 호출 *전에* 영속된 결제 레코드를 기준으로 추적된다 — 호출 중 중단되어도 그 결제건은 복구 경로로 다시 다뤄진다.

**주문과의 정합성·보상**
- 결제 결과와 주문 상태는 어긋날 수 없다 — 결제 승인은 주문 결제완료에, 결제 실패는 주문 결제실패에 대응한다.
- 결제가 실패로 확정될 때 주문 보상(차감 재고 원복 · 사용 쿠폰 원복) 이 함께 따른다 — 보상과 주문 상태 전이는 결제 정산과 같은 경계 안에서 한 번만 일어난다.
- 주문 상태 전이와 자원 보상의 규칙 자체는 Order(및 Product · Coupon) 도메인이 소유하며, 본 도메인은 결과 확정 시점에 그 전이·보상을 조율한다.

**보안**
- 카드 번호는 어떤 응답·로그에도 노출되지 않는다.

> 데이터 구조 결정은 [docs/domain/payment/logical-model.md](./logical-model.md) 를 참고한다.

## 6. 비기능 요구사항

- 외부 PG 호출에는 연결(connect) 과 응답(read) 타임아웃을 분리 설정한다 — 외부 지연이 호출 스레드·커넥션을 무한 점유하지 않는다.
- 회로 차단기로 외부 PG 의 반복 실패를 감지해 호출을 차단한다 — 차단(Open) 동안의 요청은 외부를 호출하지 않고 즉시 Fallback(처리 중 응답) 으로 처리한다.
- 외부 PG 가 다운·지연이어도 결제 요청 API 는 스레드 점유 없이 즉시 응답한다 — 외부 장애가 내부 응답성으로 전파되지 않는다.
- 콜백을 받지 못한 처리 중 결제는 주기적 폴링으로 유한 시간 안에 최종 상태로 수렴한다.
- 같은 주문에 대한 동시 결제 요청에도 진행 중 결제는 1건을 넘지 않는다 — 1주문 1결제 약속이 동시성에서 깨지지 않는다.
- 콜백과 폴링이 동시에 같은 결제를 정산해도 결과는 한 번만 확정된다 — 멱등 약속이 동시성에서 깨지지 않는다.
- 카드 번호 등 결제 민감 정보는 로깅·메트릭·응답에서 마스킹되거나 제외된다.
- 외부 호출 결과, 회로 차단기 상태 전이, 폴링 복구 결과는 관측 가능하게 로깅·계측한다 — 단, 카드 번호는 남기지 않는다.
- 모든 응답은 표준 응답 봉투(`ApiResponse<T>`) 로 감싼다.
- 시각 처리(요청·승인·취소 시각, 폴링 주기 비교) 는 `Asia/Seoul` 기준으로 일관되게 처리한다.
