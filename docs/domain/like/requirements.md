# 좋아요(Like) 도메인 요구사항 명세

> **상태**: **v0.2 초안**  
> **작성일**: 2026-05-19  
> **변경 이력**:
> - v0.2 (2026-05-20): 도메인 디렉토리 식별자를 `likes` → `like` 단수형으로 통일(저장소의 도메인 패키지 컨벤션과 정렬). 본문 경로 참조를 동기화한다. URL 리소스 컨벤션(`/api/v1/.../likes`) 은 API 스펙에서 별도로 다룬다.
> - v0.1 (2026-05-19): 최초 초안. 회원과 상품의 관계로서의 좋아요를 정의하고, 등록·취소를 분리된 두 유스케이스(멱등 수렴) 와 내 좋아요 목록 조회를 다룬다. `product` 도메인 명세 v0.2 에서 "별도 도메인" 으로 분리하기로 합의한 결과를 받는다.

---

## 1. 유비쿼터스 언어

| 용어 | 의미 |
|---|---|
| **좋아요(Like)** | 회원이 상품에 표시하는 관심. **회원과 상품의 관계** 이며 `(회원, 상품)` 쌍으로 유일하다. 좋아요한 시각을 가지며, 시각은 내 좋아요 목록의 정렬 기준으로만 사용되고 외부 응답에는 노출되지 않는다. 영문 표기는 `Like`. |
| **회원(User)** | 좋아요의 주체. User 도메인이 정의한 식별·인증 절차를 그대로 사용한다. 비회원은 본 도메인의 어떤 흐름에도 진입할 수 없다. |
| **상품(Product)** | 좋아요의 대상. Product 도메인이 정의하며, 본 도메인은 그 존재 여부와 요약 정보를 외부 의존으로 사용한다. |
| **헤더 인증** | User 도메인이 정의한 `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 기반 식별 절차. 본 도메인의 모든 유스케이스 진입 전 선행한다. |
| **멱등 수렴** | 같은 요청을 N회 보내도 최종 상태와 응답이 동일한 성질. 본 도메인의 좋아요 등록·취소는 모두 이 약속을 따른다. |
| **내 좋아요 목록** | 인증된 회원이 좋아요한 상품들의 페이지 단위 결과. 좋아요한 시각 내림차순으로 정렬된다. |
| **페이지(Page)** | 내 좋아요 목록의 분할 단위. `page` (0부터) 와 `size` 두 값으로 식별되는 offset 페이징을 사용한다. 미지정 시 `page=0` / `size=20` 이 적용된다. |
| **상품 요약** | 내 좋아요 목록의 각 항목이 담는 상품 정보 — 상품 식별자 · 이름 · 가격 · 좋아요 수 · 브랜드 요약. |

---

## 2. 유저 스토리

- 회원은 상품에 좋아요를 등록할 수 있다.
- 회원은 상품에 등록한 좋아요를 취소할 수 있다.
- 회원은 같은 등록·취소 요청을 여러 번 보내도 결과가 흔들리지 않음을 기대할 수 있다.
- 회원은 자신이 좋아요한 상품 목록을 조회할 수 있다 — 최근에 좋아요한 것부터 본다.
- 회원은 자기 자신의 좋아요 목록만 조회할 수 있다 — 타인의 목록을 조회할 수 없다.

---

## 3. 기능 요구사항

- 회원이 인증 헤더와 함께 좋아요 등록을 요청하면, 신규로 등록되거나 이미 등록된 상태가 유지되며 동일한 성공 응답을 받는다.
- 회원이 인증 헤더와 함께 좋아요 취소를 요청하면, 실제로 취소되거나 좋아요가 없는 상태가 유지되며 동일한 성공 응답을 받는다.
- 좋아요 등록·취소 응답은 빈 페이로드(`data: null`) 다 — 토글 결과 상태는 응답으로 돌려주지 않는다.
- 회원이 존재하지 않는 상품에 좋아요 등록 또는 취소를 요청하면 "찾을 수 없음" 실패 응답을 받는다 — 등록·취소 어느 쪽도 무동작 멱등으로 흡수되지 않는다.
- 인증되지 않은 요청은 좋아요 등록·취소·내 목록 조회 어느 흐름에도 진입할 수 없다.
- 회원이 자신의 식별자로 좋아요 목록을 요청하면, 자신이 좋아요한 상품들의 요약을 좋아요한 시각 내림차순으로 페이지 단위로 받는다.
- 회원이 페이지 번호·페이지 크기를 지정하지 않으면 기본값(`page=0` / `size=20`) 으로 응답된다.
- 회원이 페이지 번호 또는 페이지 크기를 허용 범위 밖으로 보내면 실패 응답을 받는다.
- 회원이 인증된 자신과 다른 사용자 식별자로 좋아요 목록을 요청하면 권한 실패 응답을 받는다.
- 좋아요 목록 응답에는 좋아요한 시각이 포함되지 않는다 — 시각은 정렬 기준으로만 사용된다.

## 4. 유스케이스 흐름

### 공통 사전 단계 — 헤더 인증

UC-1, UC-2, UC-3 의 Main Flow 진입 전에 항상 선행한다.

1. 시스템은 `X-Loopers-LoginId` 와 `X-Loopers-LoginPw` 두 헤더의 존재를 확인한다.
2. 시스템은 LoginId 로 회원을 조회하고, 저장된 비밀번호와 헤더 값의 일치를 확인한다.
3. 두 검사가 모두 통과하면 해당 회원을 컨텍스트로 두고 UC 본 흐름으로 진입한다.

**Exception**
- **E0** (사전 단계 어디서든): 헤더 누락 / 회원 없음 / 비밀번호 불일치 중 하나라도 발생 → `UNAUTHORIZED` (401), 상태 변경 없음. 어느 원인인지 응답에서 구분하지 않는다.

### UC-1. 좋아요 등록

> 공통 사전 단계(헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 상품 식별자와 함께 좋아요 등록을 요청한다.
2. 시스템은 해당 상품의 존재를 확인한다.
3. 시스템은 `(회원, 상품)` 쌍의 좋아요가 이미 존재하는지 확인한다.
4. 좋아요가 존재하지 않으면 새로 등록한다.
5. 시스템은 `200 OK` 와 빈 페이로드(`data: null`) 를 응답한다.

**Alternate Flow — 이미 등록된 상태** (3 → 4a)
- 4a. 시스템은 추가 등록 없이 현재 상태를 유지한다 (멱등 수렴, 동일 성공 응답으로 진행).

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (2): 상품 식별자가 존재하지 않음 → `PRODUCT_NOT_FOUND` (404), 등록 없음.

### UC-2. 좋아요 취소

> 공통 사전 단계(헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 상품 식별자와 함께 좋아요 취소를 요청한다.
2. 시스템은 해당 상품의 존재를 확인한다.
3. 시스템은 `(회원, 상품)` 쌍의 좋아요가 존재하는지 확인한다.
4. 좋아요가 존재하면 제거한다.
5. 시스템은 `200 OK` 와 빈 페이로드(`data: null`) 를 응답한다.

**Alternate Flow — 좋아요가 없는 상태** (3 → 4a)
- 4a. 시스템은 제거 없이 현재 상태를 유지한다 (멱등 수렴, 동일 성공 응답으로 진행).

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (2): 상품 식별자가 존재하지 않음 → `PRODUCT_NOT_FOUND` (404), 취소 없음.

### UC-3. 내 좋아요 목록 조회

> 공통 사전 단계(헤더 인증) 를 먼저 통과한다.

**Main Flow**
1. 회원이 자신의 사용자 식별자와 페이지 번호(선택) · 페이지 크기(선택) 를 담아 좋아요 목록을 요청한다.
2. 시스템은 요청의 사용자 식별자가 인증된 회원의 식별자와 일치하는지 확인한다.
3. 시스템은 페이지 번호 · 페이지 크기가 허용 범위 안에 있는지 검증한다.
4. 시스템은 미지정 입력을 기본값(`page=0` / `size=20`) 으로 대체한다.
5. 시스템은 해당 회원이 좋아요한 상품들을 좋아요한 시각 내림차순으로 페이지 단위로 응답한다.
6. 응답 각 항목은 상품 요약(상품 식별자 · 이름 · 가격 · 좋아요 수 · 브랜드 요약) 을 포함하며, 좋아요한 시각은 포함하지 않는다.

**Alternate Flow — 좋아요한 상품이 하나도 없음** (5 → 5a)
- 5a. 시스템은 빈 목록과 페이지 메타로 응답한다 (실패가 아니다).

**Exception Flow**
- E0 에 위임 (헤더 인증 실패).
- **E1** (2): 요청의 사용자 식별자가 인증된 회원과 다름 → `FORBIDDEN` (403), 상태 변경 없음.
- **E2** (3): 페이지 번호가 음수이거나 페이지 크기가 허용 범위를 벗어남 → `BAD_REQUEST` (400), 상태 변경 없음.

## 5. 도메인 규칙 / 불변식

- 좋아요는 `(회원, 상품)` 쌍으로 유일하다 — 같은 쌍의 좋아요는 1개를 넘지 않는다.
- 좋아요 등록은 멱등 수렴한다 — 이미 존재하는 좋아요는 추가되지 않으며, 응답은 신규 등록과 동일하다.
- 좋아요 취소는 멱등 수렴한다 — 존재하지 않는 좋아요의 취소도 무동작으로 성공한다.
- 좋아요 등록·취소 대상 상품이 존재하지 않으면 좋아요는 만들어지지 않고 `PRODUCT_NOT_FOUND` 로 분기된다 — 무동작 멱등으로 흡수하지 않는다.
- 좋아요 등록 · 취소 · 내 목록 조회는 모두 인증된 회원에게만 허용된다 — 비회원/미인증 요청은 본 도메인 흐름에 진입할 수 없다.
- 회원은 자기 자신의 좋아요 목록만 조회할 수 있다 — 타인의 좋아요 목록은 본 도메인을 통해 노출되지 않는다.
- 좋아요는 좋아요한 시각을 가진다. 시각은 내 좋아요 목록의 정렬 기준이며, 외부 응답에는 노출되지 않는다.
- 좋아요 등록 · 취소의 성공 응답은 빈 페이로드(`data: null`) 다 — 토글 결과 상태(`liked`, `likeCount`) 는 본 도메인 응답으로 돌려주지 않는다.
- 내 좋아요 목록의 항목은 상품 요약(식별자 · 이름 · 가격 · 좋아요 수 · 브랜드 요약) 으로 구성된다 — 좋아요한 시각은 항목에 포함되지 않는다.
- 본 도메인은 `product` 도메인의 좋아요 수(`likeCount`) 응답이 본 도메인 행 집합과 정합하도록 보장할 책임을 진다.
- **TBD**: 좋아요 수의 진실의 원천 — 상품 행의 카운트 칼럼(토글 시 증감) 인지, 좋아요 행의 `COUNT(*)` 집계인지. `product` 도메인 명세와 공유되는 미해결 결정이며 `logical-model.md` 에서 결정한다.
- **TBD**: 좋아요한 상품이 이후 판매 중지/삭제될 경우 내 좋아요 목록 노출 정책. `product` 도메인의 판매 상태 결정과 연동된다.

## 6. 비기능 요구사항

- 모든 응답은 표준 응답 봉투(공통 메타 + data) 로 감싼다. 빈 페이로드 성공도 `data: null` 형태를 유지한다.
- 좋아요 등록 · 취소는 트랜잭션 경계 안에서 수행되며, 동일 `(회원, 상품)` 쌍에 대한 동시 요청에서도 유일성 약속을 위반하지 않는다.
- 내 좋아요 목록 조회는 페이지 단위 응답을 보장하며, 페이지 크기에는 합리적 상한이 적용된다.
