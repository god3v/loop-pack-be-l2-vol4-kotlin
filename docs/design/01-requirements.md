# 요구사항 통합

> 4개 도메인의 요구사항 명세를 한 자리에서 일람한다. 본 문서는 도메인의 의도·범위·핵심 약속까지만 다루고, 구체 흐름·예외 분기·도메인 규칙·비기능은 각 도메인의 `requirements.md` 를 본다.

## 도메인 일람

| 도메인 | 한 줄 설명 | 명세 |
|---|---|---|
| 사용자(User) | 회원 가입·헤더 인증·내 정보 조회·비밀번호 변경 | [docs/domain/user/requirements.md](../domain/user/requirements.md) |
| 상품(Product) | 카탈로그 조회 + 관리자 채널 (브랜드·상품 CRUD) | [docs/domain/product/requirements.md](../domain/product/requirements.md) |
| 좋아요(Like) | 회원이 상품에 표시하는 관심의 등록·취소, 내 좋아요 목록 | [docs/domain/like/requirements.md](../domain/like/requirements.md) |
| 주문(Order) | 주문 생성·결제, 회원·운영자 시각 조회 | [docs/domain/order/requirements.md](../domain/order/requirements.md) |

---

## 사용자(User)

> 회원의 가입과 본인 확인을 담는다. 모든 인증된 흐름은 `X-Loopers-LoginId` / `X-Loopers-LoginPw` 헤더 기반 식별을 사전 단계로 둔다.

**유스케이스**
- UC-1. 회원 가입
- UC-2. 내 정보 조회
- UC-3. 비밀번호 수정

**핵심 약속**
- `LoginId` · `Password` · `Email` 은 값 객체 — 형식 검증은 생성 시점에 수행한다.
- 비밀번호 RULE(길이 8~16 · 영문/숫자/특수 3카테고리 각 1자 이상 · 생년월일 미포함) 은 가입·변경에 동일하게 적용된다.
- 헤더 인증 실패의 원인(헤더 누락 / 계정 없음 / 비밀번호 불일치) 은 응답에서 구분하지 않는다 — 계정 존재 여부를 외부에 노출하지 않기 위함이다.
- 비밀번호는 평문도 해시도 어떤 응답·로그에도 노출되지 않는다.

---

## 상품(Product)

> 카탈로그 조회 + 관리자 채널을 함께 담는다. 회원 카탈로그는 인증 없이 열람되고, 관리자 채널은 관리자 인증을 사전 단계로 둔다. 좋아요 수·`likedByMe` 는 외부 `like` 도메인의 결과를 본 도메인 응답이 반영한다.

**유스케이스**

회원 카탈로그:
- UC-1. 상품 목록 조회
- UC-2. 상품 상세 조회
- UC-3. 단일 브랜드 조회

관리자 채널:
- UC-4. 브랜드 목록 조회
- UC-5. 브랜드 상세 조회
- UC-6. 브랜드 등록
- UC-7. 브랜드 정보 수정
- UC-8. 브랜드 삭제
- UC-9. 상품 목록 조회
- UC-10. 상품 상세 조회
- UC-11. 상품 등록
- UC-12. 상품 정보 수정
- UC-13. 상품 삭제

**핵심 약속**
- 상품은 정확히 하나의 브랜드에 속한다. 가격·좋아요 수는 음수가 될 수 없다.
- 정렬 옵션은 `latest` · `price_asc` · `likes_desc` 세 값으로 한정된다. 미지정 시 `latest`.
- 페이지 기본값은 `page=0` / `size=20`. 허용 범위 밖 입력은 실패 분기로 분리된다.
- 회원 카탈로그 조회는 인증을 필수로 하지 않는다. 관리자 채널은 관리자 인증을 사전 단계로 둔다 — 인증의 구체 메커니즘은 본 명세 범위 밖이다.
- 좋아요 등록/취소의 책임은 `like` 도메인에 있고, 본 도메인은 그 결과를 카탈로그 응답에 반영한다.
- 삭제는 soft delete (삭제 마크). 브랜드 삭제는 소속 상품을 카스케이드로 삭제 마크 처리한다 (단일 트랜잭션, 부분 실패 시 전체 롤백).
- 상품의 브랜드는 등록 이후 변경 불가하다. 상품 이름은 같은 브랜드 안에서 유일하다.
- 상품 삭제 시 좋아요 행은 유지된다 — 내 좋아요 목록에서는 가려진다.
- 판매 상태(`SalesStatus`) 는 관리자 응답에 포함된다.

## 좋아요(Like)

> 회원과 상품의 관계로서 좋아요를 정의한다. `(회원, 상품)` 쌍으로 유일하며, 등록·취소가 모두 멱등 수렴한다.

**유스케이스**
- UC-1. 좋아요 등록
- UC-2. 좋아요 취소
- UC-3. 내 좋아요 목록 조회

**핵심 약속**
- 좋아요는 `(userId, productId)` 쌍으로 유일하다 — 같은 쌍의 좋아요는 1개를 넘지 않는다.
- 등록·취소는 멱등 수렴한다 — 이미 등록된 상태의 재등록, 좋아요 없음 상태의 취소 모두 무동작 성공이다.
- 등록·취소 대상 상품이 존재하지 않으면 `PRODUCT_NOT_FOUND` 로 분기된다 — 무동작 멱등으로 흡수하지 않는다.
- 모든 흐름은 인증된 회원에게만 허용된다. 회원은 자기 자신의 좋아요 목록만 조회할 수 있다.
- 등록·취소 응답은 빈 페이로드(`data: null`) 다 — 토글 결과 상태는 본 도메인이 반환하지 않는다.

## 주문(Order)

> 주문 생성·결제 흐름과 조회를 담는다.

**유스케이스**

회원 시각:
- UC-1. 주문 생성
- UC-2. 내 주문 목록 조회
- UC-3. 단일 주문 상세 조회

운영자 시각:
- UC-4. 어드민 주문 목록 조회
- UC-5. 어드민 단일 주문 상세 조회

**핵심 약속**
- 주문은 1개 이상의 라인 아이템을 가진다. 각 라인은 주문 시점의 가격·상품명을 스냅샷으로 박아 보관한다.
- 주문 상태는 `결제 대기(PAYMENT_PENDING)` / `결제 완료(PAID)` / `취소(CANCELLED)` 셋으로 한정된다. 결제 실패는 별도 상태가 아니라 `CANCELLED` 로 귀결된다. 사용자 발기 취소·배송 등은 본 명세 범위 밖이다.
- 주문 생성은 멱등 키로 수렴한다 — 같은 회원의 같은 키 재요청은 신규를 만들지 않고 기존 응답을 그대로 돌려준다.
- 합의 순서: 상품 존재 확인 → 재고 차감 → (쿠폰 적용) → 외부 결제. 후속 단계의 실패는 앞 단계를 명시적으로 되돌리는 보상 책임을 본 도메인이 진다.
- 회원은 자기 자신의 주문만 본다 — 단일 조회의 타인 식별자는 `FORBIDDEN`, 존재하지 않는 식별자는 `ORDER_NOT_FOUND` 로 의미를 구분한다.
- 운영자에게는 본인 제약이 적용되지 않고, 응답에 운영 메타(회원 식별자·표시명·외부 결제 트랜잭션 식별자·결과 코드) 가 추가된다.
