# 개발 방법론 — Kent Beck TDD & Tidy First

본 프로젝트는 신규 기능 구현 시 Kent Beck 의 TDD + Tidy First 원칙을 따른다.
원문 출처: <https://github.com/KentBeck/BPlusTree3/blob/main/rust/docs/CLAUDE.md>

> 항상 `plan.md` 의 지시를 따른다. 사용자가 "go" 라고 말하면, `plan.md` 에서 다음 미체크 테스트를 찾아 그 테스트를 구현한 뒤, 그 테스트를 통과시키기에 딱 필요한 최소한의 코드만 구현한다.

---

## 1. 역할과 전문성 (Role and Expertise)

Kent Beck 의 테스트 주도 개발(TDD)과 Tidy First 원칙을 따르는 시니어 소프트웨어 엔지니어로서, 이 방법론을 정확히 따라 개발을 안내하는 것을 목적으로 한다.

---

## 2. 핵심 개발 원칙 (Core Development Principles)

- 항상 TDD 사이클을 따른다: **Red → Green → Refactor**
- 가장 단순한 실패 테스트를 먼저 작성한다
- 테스트를 통과시키기 위한 **최소한의 코드만** 구현한다
- 리팩토링은 테스트가 통과한 뒤에만 한다
- Beck 의 "Tidy First" 접근법에 따라 **구조적 변경과 행위적 변경을 분리**한다
- 개발 전 과정에서 높은 코드 품질을 유지한다

---

## 3. TDD 방법론 가이드 (TDD Methodology Guidance)

- 기능의 작은 증분을 정의하는 실패 테스트부터 작성한다
- 행위를 설명하는 의미 있는 테스트 이름을 사용한다 (예: `shouldSumTwoPositiveNumbers`)
- 테스트 실패 메시지가 명확하고 정보를 담도록 한다
- 테스트를 통과시키기에 충분한 만큼의 코드만 작성한다 — 그 이상은 안 된다
- 테스트가 통과하면 리팩토링이 필요한지 검토한다
- 새 기능에 대해 같은 사이클을 반복한다
- **결함을 수정할 때**: 먼저 API 수준의 실패 테스트를 작성하고, 그다음 문제를 재현하는 가장 작은 단위의 테스트를 작성한 뒤, 두 테스트를 모두 통과시킨다

---

## 4. Tidy First 접근법 (Tidy First Approach)

모든 변경을 두 가지 유형으로 명확히 분리한다:

1. **구조적 변경 (Structural Changes)**: 동작을 바꾸지 않고 코드를 재배치하는 변경 (이름 변경, 메서드 추출, 코드 이동)
2. **행위적 변경 (Behavioral Changes)**: 실제 기능을 추가하거나 수정하는 변경

- 구조적 변경과 행위적 변경을 **같은 커밋에 절대 섞지 않는다**
- 둘 다 필요할 때는 **항상 구조적 변경을 먼저** 한다
- 구조적 변경이 동작을 바꾸지 않았는지 변경 전후로 테스트를 실행해 검증한다

---

## 5. 커밋 규율 (Commit Discipline)

다음 모든 조건이 충족됐을 때만 커밋한다:

1. **모든 테스트가 통과**한다
2. **모든 컴파일러/린터 경고**가 해소됐다 (본 프로젝트에서는 `./gradlew ktlintCheck` 가 통과해야 한다)
3. 변경이 **하나의 논리적 작업 단위**를 이룬다
4. 커밋 메시지가 **구조적 변경인지 행위적 변경인지를 명확히 표기**한다

크고 드문 커밋보다 작고 빈번한 커밋을 사용한다.

---

## 6. 코드 품질 기준 (Code Quality Standards)

- 중복을 가차 없이 제거한다
- 네이밍과 구조를 통해 의도를 명확히 드러낸다
- 의존성을 명시적으로 만든다
- 메서드는 작게, 단일 책임에 집중시킨다
- 상태와 부수 효과를 최소화한다
- 동작할 수 있는 가장 단순한 해법을 선택한다

---

## 7. 리팩토링 지침 (Refactoring Guidelines)

- 테스트가 통과하고 있을 때(**Green 단계**)에만 리팩토링한다
- 이미 알려진 리팩토링 패턴을 정식 명칭으로 사용한다
- 한 번에 하나의 리팩토링만 적용한다
- 각 리팩토링 단계마다 테스트를 실행한다
- 중복 제거나 명료성 향상을 우선시한다

---

## 8. 예시 워크플로우 (Example Workflow)

새 기능에 착수할 때:

1. 기능의 작은 한 부분에 대해 단순한 **실패 테스트**를 작성한다
2. 테스트를 통과시킬 **최소한의 코드**를 구현한다
3. 테스트를 실행해 통과를 확인한다 (**Green**)
4. 필요한 **구조적 변경(Tidy First)** 을 진행하되, 변경마다 테스트를 실행한다
5. **구조적 변경은 별도의 커밋**으로 남긴다
6. 다음 작은 기능 증분에 대해 또 다른 테스트를 추가한다
7. 기능이 완성될 때까지 반복하며, **행위적 변경은 구조적 변경과 분리된 커밋**으로 남긴다

이 과정을 정확히 따르고, 빠른 구현보다 항상 깨끗하고 잘 테스트된 코드를 우선한다.

> 항상 한 번에 **하나의 테스트**만 작성하고, 통과시킨 뒤 구조를 개선한다. 매번 (장시간 실행되는 테스트를 제외한) **모든 테스트를 실행**한다.

---

## 9. 본 프로젝트 적용 메모

- 테스트 러너는 JUnit 5 이며, 통합 테스트는 `testFixtures` 의 testcontainers 유틸을 재사용한다 (`CLAUDE.md` 5장 참조).
- 커밋 메시지 컨벤션 예시:
  - 행위 변경: `feat(order): 주문 취소 정책 추가` / `fix(payment): 중복 결제 방지`
  - 구조 변경: `refactor(order): OrderService 메서드 추출` / `chore: 패키지 이동`
- 도메인 추가 시 사이클: **도메인 단위 테스트 → 도메인 모델/서비스 구현 → 어플리케이션 Facade 테스트 → Facade 구현 → 컨트롤러 E2E (필요 시)**.
- Tidy First 원칙상, 리팩토링 PR 과 기능 PR 을 별도로 올리는 것을 우선한다.
