# 요구사항 명세 템플릿

> 새 도메인의 요구사항 명세를 작성할 때 본 템플릿을 복사해 사용한다.
> 위치 예: `docs/domain/<도메인>/requirements.md`
> 본 문서는 **섹션 골격** 과 **작성 규약** 두 부분으로 구성된다.

---

## 0. 위치와 역할

| 문서 | 답하는 질문 | 산출 스킬 |
|---|---|---|
| `requirements.md` | **무엇이 일어나는가 / 어떻게 흘러가는가** | `/requirements` (본 템플릿) |
| `logical-model.md` | **어떤 데이터 구조로 표현하는가** | (별도 — 사용자 작성) |
| `api-spec.md` | **어떤 인터페이스로 노출하는가** | `/api-spec` |
| `plan.md` | **어떤 순서로 구현하는가** (TDD) | `/tdd` |

`requirements.md` 는 다른 문서의 **입력 원천** 이다. 구현 디테일(클래스명, 메서드 시그니처, 알고리즘) 은 여기서 다루지 않는다.

## 2. 작성 규약

### 2.1 어휘는 §1 에서 고정한다
- §1 유비쿼터스 언어에 정의한 용어 표기를 §2~§5 전체에서 일관되게 사용한다.
- 같은 개념을 다른 단어로 부르지 않는다 (예: "찜" 과 "관심상품" 을 섞어 쓰지 않는다).

### 2.2 유저 스토리는 능동·외부 관점
- "**사용자는 ~할 수 있다**" 형식. 시스템 내부 동작을 적지 않는다.
- 시나리오는 한 줄 자연어 한 문장. 조건 분기가 있으면 별도 문장으로 분리한다.

### 2.3 기능 요구사항은 관찰 가능한 동작만
- "저장한다"(내부) 가 아니라 "화면에 반영된다"(외부 관찰).
- 시스템 구성요소·DB·캐시·라이브러리 단어가 등장하면 그 문장은 §5 (도메인 규칙) 또는 §6 (비기능 요구사항) 으로 옮긴다.

### 2.4 유스케이스 흐름의 형식
- **Main Flow** 는 골든 패스 — 동사로 시작하는 한 문장씩 번호를 매긴다.
- **Alternate Flow** 는 정상 범위 내 분기 — 헤더에 `({원본 단계} → {분기 단계})` 로 어디서 갈라지는지 명시한다.
- **Exception Flow** 는 오류 분기 — `E{N} ({어느 단계})` 형식으로 시작하고, 결과 응답과 상태 영향(상태 변경 없음 / 부분 실패 / 멱등 수렴) 을 명시한다.
- 각 단계 안에는 클래스명·메서드명·SQL 을 적지 않는다.

### 2.5 도메인 규칙은 사실로만
- 검증/저장/계산의 **방법** 이 아니라 **사실** 만 적는다.
  - ✅ "찜 수는 음수가 될 수 없다."
  - ❌ "LikeService 가 count -= 1 을 호출하기 전 0 인지 검사한다."

### 2.5a 도메인 규칙(§5) 과 비기능 요구사항(§6) 의 구분
- **§5 도메인 규칙** — 도메인 자체의 의미적 약속 (불변식, 책임 분리, 멱등성).
- **§6 비기능 요구사항** — 도메인과 무관한 시스템 차원 요구 (성능 목표, 응답 포맷, 로깅 정책, 시간대).
- 헷갈리면 "이 규칙이 깨지면 도메인이 거짓이 되는가, 운영 품질이 떨어지는가" 로 가른다 — 전자는 §5, 후자는 §6.

### 2.6 데이터 모델은 별도 문서
- 컬럼/타입/제약 같은 데이터 구조 결정은 본 문서가 아니라 `logical-model.md` 가 다룬다.
- §5 말미에 "데이터 구조 결정은 `logical-model.md` 를 참고한다" 한 줄로 위임한다.

### 2.7 임의 추정 금지
- 사용자가 명시하지 않은 결정은 추정해 채우지 않는다.
- 미정 항목은 본문에 `**TBD**: {질문}` 으로 자리만 남기고, 문서 마지막 `# 미해결 결정 사항` 섹션에 질문 목록을 모은다.

### 2.8 톤
- 본문은 한국어 `~다` 종결로 통일한다.
- 변경 이력은 최신 버전이 위에 오도록 역순 누적한다. 보강·수정 시 본문을 덮어쓰지 말고 새 버전 줄을 추가한다.
- 본 명세는 "정답" 이 아니라 합의 결과를 적은 문서다 — 결정되지 않은 항목을 결정된 것처럼 적지 않는다. 의도·책임·경계가 코드 디테일보다 앞선다.

## 3. 작성 체크리스트 (확정 전 확인)

- [ ] §1 에서 도메인 핵심 용어가 모두 정의되었고, §2~§6 가 그 어휘로 일관되게 쓰여 있다
- [ ] §2 유저 스토리가 "사용자는 ~" 능동 문장으로만 적혀 있다
- [ ] §3 기능 요구사항에 시스템 내부 단어(DB/캐시/메서드명) 가 섞이지 않았다
- [ ] §4 의 모든 유스케이스가 Main / Alternate / Exception 세 갈래로 다 채워져 있다
- [ ] 각 Exception 항목에 결과 응답과 상태 영향이 명시되어 있다
- [ ] §5 가 "사실" 만 진술하고 "방법" 을 진술하지 않는다
- [ ] §5 말미에 `logical-model.md` 위임 줄이 있다
- [ ] §6 비기능 요구사항이 §5 도메인 규칙과 책임이 섞이지 않았다
- [ ] 구현 디테일(클래스명/메서드명/SQL) 이 본문에 섞이지 않았다
- [ ] 미정 항목은 `TBD: ...` 로 표기되고 마지막 "미해결 결정 사항" 섹션에 모여 있다
- [ ] 변경 이력에 현재 버전 줄이 추가되었다
