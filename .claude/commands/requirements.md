# /requirements — 요구사항 명세 작성

> 본 커맨드는 [docs/guideline/requirements-template.md](../../docs/guideline/requirements-template.md) 의 규약을 따른다.
> 본 명세 작성은 단순 받아쓰기가 아니라 **문제 분석 → 합의 → 본문 작성** 흐름을 거친다.
> 인자: `$ARGUMENTS` — 도메인 모델명 (예: `user`, `order`, `product`).
> 출력 파일: `docs/domain/$ARGUMENTS/requirements.md`

---

## 분석 자세 (Mindset)

요구사항을 다룰 때 다음 4단계를 반드시 거친다. 결과는 본문 작성 **전에** 사용자에게 제시하고 합의를 받는다.

### 1단계. 요구사항을 그대로 믿지 않고 문제 상황으로 재해석한다
- 문장 정리에서 끝내지 않는다 — "**지금 어떤 문제가 있고, 왜 해결하려는가**" 로 다시 본다.
- 3관점으로 분리해 정리한다:
  - **사용자 관점** — 사용자가 겪는 불편 또는 기대
  - **비즈니스 관점** — 도메인/조직이 얻거나 잃는 것
  - **시스템 관점** — 일관성·정합성·확장성·신뢰성 측면의 문제
- 예: "주문 실패 시 결제를 취소한다" → "결제 성공/실패와 주문 상태가 어긋나지 않도록 일관성을 유지하는 문제"

### 2단계. 애매한 부분을 숨기지 않고 드러낸다
- 임의 결정·추측 금지. 결정되지 않은 부분을 **명시적으로 나열**한다.
- 다음 유형을 반드시 점검한다:
  - **정책 질문** — 기준 시점, 성공/실패 조건, 예외 처리 규칙
  - **경계 질문** — 어디까지가 한 책임인가, 어디서 분리되는가
  - **확장 질문** — 향후 바뀔 가능성이 있는 항목

### 3단계. 명확화 질문은 개발자가 답하기 쉬운 형태로 제시한다
- 질문은 **우선순위**를 매긴다 (중요한 것부터).
- 선택지가 있을 때는 **옵션 + 영향도** 를 함께 제시한다.
- 예시 형식:
  ```
  Q1. (정책) 주문 실패 시 결제 취소 시점은?
   - A) 결제 직후 즉시 취소 → 구현 단순, 사용자에게 빠른 확정 응답. 확장성 낮음.
   - B) 비동기 보상 트랜잭션  → 구조 복잡, 일시적 불일치 허용. 확장·보상 처리 유리.
  ```

### 4단계. 개념 모델부터 잡는다 (구현 전 설계 사고 정렬)
- 코드/기술 얘기 전에 다음을 먼저 정의한다:
  - **액터** — 사용자, 외부 시스템
  - **핵심 도메인** — 본 명세의 주체
  - **보조/외부 시스템** — 외부 의존, 비동기 채널, 캐시 등
- 이 단계의 산출물은 §1 유비쿼터스 언어와 §5 도메인 규칙의 입력이 된다.

## 톤 & 스타일

- **설계 리뷰 톤** — 강의처럼 설명하지 않는다. "이게 정답이다" 가 아니라 "이런 선택지가 있고, 각각의 영향은 …" 으로 풀어낸다.
- **의도 · 책임 · 경계 중심** — 코드 디테일보다 "왜 그렇게 나뉘는가" 를 먼저 다룬다.
- **결정되지 않은 것을 결정된 것처럼 적지 않는다** — 추측은 질문으로 환원한다.
- 본문(문서) 톤은 한국어 `~다` 종결, 분석·질문 대화 톤은 사용자에게 존댓말을 유지한다.

## 실행 절차

### 1) 입력 검증
- `$ARGUMENTS` 가 비어 있으면 즉시 중단하고 도메인 모델명을 묻는다.
- 도메인 모델명에는 공백/슬래시를 허용하지 않는다.

### 2) 자료 로드
1. `docs/guideline/requirements-template.md` 를 읽어 **섹션 골격(`§1`)** 과 **작성 규약(`§2`)** 을 적재한다.
2. `docs/domain/$ARGUMENTS/requirements.md` 가 이미 있으면 그 내용을 읽어 **덮어쓰지 말고 보강·수정** 한다. 새 변경은 상단 변경 이력에 한 줄로 추가한다.
3. `docs/domain/$ARGUMENTS/logical-model.md` 가 있으면 함께 읽어 **§1 용어와 §5 도메인 규칙의 정합성 점검 컨텍스트** 로 둔다. 없으면 무시한다.

### 3) 분석 & 합의 (본문 작성 전)
"분석 자세" 4단계를 수행하고, 산출물을 **본문 작성보다 먼저** 사용자에게 제시한다:

1. **문제 재해석** — 사용자/비즈니스/시스템 3관점 요약 (각 1~3줄)
2. **드러난 애매함** — 정책/경계/확장 카테고리별로 결정되지 않은 항목 나열
3. **명확화 질문** — 우선순위(Q1, Q2 …) + 선택지 + 영향도 형식
4. **개념 모델 초안** — 액터 / 핵심 도메인 / 보조·외부 시스템

사용자의 응답·결정을 받은 뒤 §4 로 넘어간다.
보강 작업이면 분석 범위를 **변경분에 한정**하되, 새로운 애매함이 드러나면 동일 절차로 다룬다.
"단순 정리/오타 교정" 수준의 변경은 분석 단계를 생략할 수 있다.

### 4) 명세 작성 규약
합의된 내용을 §1~§6 본문에 녹인다. 합의되지 않은 항목은 본문에 `**TBD**: {질문}` 으로 자리만 남기고 마지막 `# 미해결 결정 사항` 섹션에 모은다.

- 템플릿 §1 의 6섹션 순서/제목을 그대로 따른다 — 새 섹션을 임의로 끼워 넣지 않는다.
  1. 유비쿼터스 언어  ← 분석 4단계 산출물(액터/핵심/보조) 이 이 섹션의 입력
  2. 유저 스토리       ← 1단계 "사용자 관점 문제" 가 입력
  3. 기능 요구사항
  4. 유스케이스 흐름
  5. 도메인 규칙 / 불변식  ← 1단계 "시스템 관점 문제" + 4단계 개념 모델이 입력
  6. 비기능 요구사항
- **§1 유비쿼터스 언어** 에서 도메인 핵심 용어를 먼저 고정하고, §2~§6 전체에서 그 어휘를 일관되게 사용한다.
- **§2 유저 스토리** 는 "사용자는 ~할 수 있다" 능동 문장만 허용한다. 시스템 내부 동작을 적지 않는다.
- **§3 기능 요구사항** 은 사용자가 관찰 가능한 동작만 적는다. DB/캐시/메서드명 같은 내부 단어가 섞이면 §5 또는 §6 으로 옮기거나 추상화한다.
- **§4 유스케이스 흐름** 은 각 유스케이스마다 **Main / Alternate / Exception** 세 갈래를 모두 채운다.
  - Main 은 골든 패스, 동사로 시작하는 한 문장씩 번호를 매긴다.
  - Alternate 헤더에는 `({원본 단계} → {분기 단계})` 로 갈라지는 지점을 명시한다.
  - Exception 은 `E{N} ({어느 단계})` 형식으로 시작하고, **결과 응답과 상태 영향** 을 명시한다.
- **§5 도메인 규칙** 은 "사실" 만 진술한다. "방법"(검증 알고리즘, 호출 순서) 은 적지 않는다.
- §5 말미에 `> 데이터 구조 결정은 [docs/domain/$ARGUMENTS/logical-model.md](./logical-model.md) 를 참고한다.` 한 줄을 반드시 둔다.
- **§6 비기능 요구사항** 은 도메인과 무관한 시스템 차원 요구(성능 목표, 응답 포맷, 로깅 정책, 시간대, 관측성) 만 다룬다. "규칙이 깨지면 도메인이 거짓이 되는가" 면 §5, "운영 품질이 떨어지는가" 면 §6.
- 구현 디테일(클래스명, 메서드 시그니처, 알고리즘, 라이브러리, SQL) 은 **금지**.

### 5) 출력 파일 구조
```
# {도메인 한국어 이름}({Domain}) 도메인 요구사항 명세

> 상태 / 작성일 / 변경 이력

## 1. 유비쿼터스 언어
## 2. 유저 스토리
## 3. 기능 요구사항
## 4. 유스케이스 흐름
## 5. 도메인 규칙 / 불변식
   > 데이터 구조 결정은 docs/domain/$ARGUMENTS/logical-model.md 를 참고한다.
## 6. 비기능 요구사항

# 미해결 결정 사항 (TBD 가 하나라도 있을 때만)
```

### 6) 작성 체크리스트 (확정 전 확인)
- [ ] 분석 4단계 산출물을 사용자에게 먼저 제시하고 합의를 받았다 (또는 보강 범위가 합의 면제 조건이다)
- [ ] §1 에 도메인 핵심 용어가 모두 정의되었고, §2~§6 가 그 어휘로 일관되게 쓰여 있다
- [ ] §2 유저 스토리가 "사용자는 ~" 능동 문장으로만 적혀 있다
- [ ] §3 기능 요구사항에 시스템 내부 단어(DB/캐시/메서드명) 가 섞이지 않았다
- [ ] §4 의 모든 유스케이스가 Main / Alternate / Exception 세 갈래로 다 채워져 있다
- [ ] 각 Exception 항목에 결과 응답과 상태 영향이 명시되어 있다
- [ ] §5 가 "사실" 만 진술하고 "방법" 을 진술하지 않는다
- [ ] §5 말미에 `logical-model.md` 위임 줄이 있다
- [ ] §6 비기능 요구사항이 §5 도메인 규칙과 책임이 섞이지 않았다
- [ ] 구현 디테일(클래스명/메서드명/SQL) 이 본문에 섞이지 않았다
- [ ] 미해결 항목은 `TBD: ...` 로 표기되고 마지막 "미해결 결정 사항" 섹션에 모여 있다
- [ ] 변경 이력에 현재 버전 줄이 추가되었다

### 7) 후속 안내
명세 작성을 마치면 사용자에게 다음 단계를 권고한다:
1. 미해결 결정 사항이 있으면 먼저 해소한다.
2. 데이터 구조 결정이 필요하면 `docs/domain/$ARGUMENTS/logical-model.md` 를 작성한다.
3. `/api-spec $ARGUMENTS` 로 API 명세를 도출한다.
4. `/tdd $ARGUMENTS` 로 plan.md 골격을 생성하고 TDD 사이클을 시작한다.

---

이제 위 절차에 따라 `$ARGUMENTS` 도메인의 `docs/domain/$ARGUMENTS/requirements.md` 를 작성/보강하라. **분석 → 합의 → 작성 순서를 건너뛰지 않는다.**
